Writing and Using Functions

▶

Functions are one of the most powerful organizational tools in Python. At their core, a function is a named, reusable block of code that performs a specific task. Instead of writing the same logic repeatedly throughout a script, you define it once as a function and call it whenever you need it. In data analytics work, this matters enormously: you might need to clean a dataset, compute a summary statistic, or format an output dozens of times across a project. Functions let you encode that logic once, test it carefully, and then trust it everywhere it is used. This topic walks through every essential aspect of writing and using functions in Python — from the basic syntax of defining one, through parameters, return values, scope, and documentation — with practical examples drawn from realistic data processing scenarios.

Defining a Function with def

Every Python function begins with the def keyword, followed by the function name, a pair of parentheses (which may contain parameters), and a colon. The code that belongs to the function — its body — must be indented consistently beneath that line. The Python community standard is four spaces per indentation level, and mixing tabs and spaces will cause errors. Here is the simplest possible function:

def greet():
    print("Hello, analyst!")

A critical concept for beginners: writing a def block does not execute the code inside it. It merely registers the function under the given name so it can be called later. Think of the definition as writing a recipe — the meal is not cooked until you actually follow the recipe. Until you call greet(), the print statement inside never runs.

Choosing descriptive function names is not just a style preference; it is a form of self-documentation. A function named remove_duplicates tells every reader — including your future self — exactly what that block of code does, without requiring them to read its body. In contrast, a function named process or func1 forces readers to inspect the implementation every time they encounter the call. In data analytics scripts that may be shared with colleagues or revisited months later, clear naming is indispensable.

def load_sales_data(filepath):
    # Opens a CSV and returns a list of rows
    with open(filepath, "r") as f:
        return f.readlines()

def remove_blank_rows(rows):
    return [row for row in rows if row.strip() != ""]

Even without reading the bodies, the purpose of each function is immediately obvious from its name alone.

Parameters and Arguments

A function that always does exactly the same thing has limited usefulness. Parameters are the mechanism that makes functions flexible: they are placeholders defined in the function signature, and they receive concrete values — called arguments — when the function is called. A function can accept one parameter, many parameters, or none at all.

def compute_average(values):
    return sum(values) / len(values)

scores = [88, 92, 75, 100, 67]
print(compute_average(scores))   # 84.4

Here values is a parameter; scores is the argument passed when the function is called. Inside the function body, values refers to whatever list was passed in.

Default parameter values make some arguments optional. When a caller does not supply a value for a parameter that has a default, Python uses the default automatically. This is particularly useful in analytics functions where sensible defaults exist for options like rounding precision or a missing-value placeholder:

def summarize(data, decimals=2):
    mean = round(sum(data) / len(data), decimals)
    minimum = round(min(data), decimals)
    maximum = round(max(data), decimals)
    return mean, minimum, maximum

print(summarize([10.5, 20.333, 15.1]))         # uses decimals=2
print(summarize([10.5, 20.333, 15.1], decimals=4))  # overrides to 4

Parameters with defaults must always appear after parameters without defaults in the function signature, otherwise Python raises a SyntaxError.

Positional arguments are matched to parameters strictly by order — the first argument goes to the first parameter, the second to the second, and so on. Keyword arguments are matched by name, regardless of position. Keyword arguments make call sites far more readable, especially when a function has several parameters:

def filter_records(data, column, threshold, above=True):
    if above:
        return [row for row in data if row[column] > threshold]
    else:
        return [row for row in data if row[column] <= threshold]

# Positional call — must remember the order:
result = filter_records(records, "revenue", 5000, True)

# Keyword call — self-documenting:
result = filter_records(data=records, column="revenue", threshold=5000, above=True)

Mixing positional and keyword arguments in a single call is allowed, but positional arguments must always come first.

Returning Values with return

The return statement sends a value back to the caller and immediately ends the function's execution. Without an explicit return, a function silently returns None — a common source of bugs when a developer forgets to include it:

def double(x):
    result = x * 2
    # Forgot to write: return result

output = double(5)
print(output)   # None  <-- not 10!

A function can return any Python object: a number, a string, a list, a dictionary, or even another function. When you need to return several related values, Python lets you return them as a tuple simply by separating them with commas on the return line:

import statistics

def descriptive_stats(data):
    mean   = statistics.mean(data)
    median = statistics.median(data)
    mode   = statistics.mode(data)
    return mean, median, mode    # returns a tuple (mean, median, mode)

prices = [12, 15, 12, 18, 20, 15, 12]
avg, med, mod = descriptive_stats(prices)   # tuple unpacking
print(f"Mean: {avg}, Median: {med}, Mode: {mod}")

Tuple unpacking on the left side of the assignment lets you assign each returned value to its own variable in a single, readable line. Capturing return values in named variables is essential for chaining processing steps: the output of one function becomes the input of the next, building a clear data pipeline:

raw       = load_sales_data("sales_2024.csv")
cleaned   = remove_blank_rows(raw)
converted = parse_numeric_fields(cleaned)
summary   = descriptive_stats(converted)

Each line is easy to read, and each intermediate result has a meaningful name.

Calling Functions

Calling a function is straightforward: write its name followed by parentheses containing any required arguments. However, a function must be defined before it is called in the script's execution order. Attempting to call a function before its def block has been executed raises a NameError:

# This raises NameError: name 'clean' is not defined
result = clean(data)

def clean(data):
    return [x for x in data if x is not None]

The conventional solution is to place all function definitions at the top of the script (or in a separate module) and put the main execution logic at the bottom. Many Python scripts wrap their top-level calls inside a if __name__ == "__main__": block for exactly this reason.

Functions can call other functions freely, enabling layered, modular pipelines. A high-level function can orchestrate several lower-level ones:

def normalize(value, min_val, max_val):
    return (value - min_val) / (max_val - min_val)

def normalize_column(data, column):
    values = [row[column] for row in data]
    min_val = min(values)
    max_val = max(values)
    for row in data:
        row[column] = normalize(row[column], min_val, max_val)
    return data

def preprocess(data):
    data = remove_blank_rows(data)
    data = normalize_column(data, "revenue")
    data = normalize_column(data, "units_sold")
    return data

preprocess calls remove_blank_rows and normalize_column; normalize_column in turn calls normalize. Each layer handles one level of abstraction, making the code easy to reason about and test independently.

Organizing Code with Modular Functions

The principle of modularity is that each function should ideally do exactly one well-defined thing. A function that reads a file, cleans the data, computes statistics, and writes a report is doing too much. If anything goes wrong — or if you later need to reuse just one of those steps — the monolithic design makes it hard. Breaking the logic into focused functions solves both problems:

def read_csv(filepath):
    """Read a CSV file and return a list of dictionaries."""
    import csv
    with open(filepath, newline="") as f:
        return list(csv.DictReader(f))

def drop_nulls(records, required_fields):
    """Remove records that are missing values in any required field."""
    return [r for r in records if all(r.get(f) for f in required_fields)]

def compute_totals(records, value_field):
    """Return the sum of a numeric field across all records."""
    return sum(float(r[value_field]) for r in records)

def write_summary(filepath, summary_dict):
    """Write a summary dictionary to a text file."""
    with open(filepath, "w") as f:
        for key, value in summary_dict.items():
            f.write(f"{key}: {value}\n")

Each function has a single responsibility. They can be mixed, matched, and reused across entirely different scripts or datasets. If your next project uses a different file format, you replace only read_csv — everything else stays the same. This is the practical power of modular design.

Grouping related functions together in a script and naming them with a consistent vocabulary also acts as inline documentation. A reader skimming the function names in a file gets a high-level map of the entire data workflow without reading a single line of function body code.

Docstrings for Function Documentation

A docstring is a string literal placed immediately after the def line, enclosed in triple quotes. It is not a comment — Python stores it as the function's __doc__ attribute, making it accessible to documentation tools, IDEs, and the built-in help() function.

def compute_growth_rate(current, previous):
    """
    Calculate the percentage growth rate between two values.

    Parameters:
        current  (float): The more recent value.
        previous (float): The baseline value to compare against.

    Returns:
        float: The growth rate as a percentage, rounded to two decimal places.
               Returns None if previous is zero to avoid division by zero.
    """
    if previous == 0:
        return None
    return round(((current - previous) / previous) * 100, 2)

A well-written docstring answers three questions: what does the function do, what does each parameter expect (including type and meaning), and what does it return. In a data analytics team context, colleagues using your function should not have to read its implementation to understand how to call it correctly — the docstring should be sufficient.

You can inspect a docstring interactively:

help(compute_growth_rate)
# or
print(compute_growth_rate.__doc__)

Popular documentation styles include Google style, NumPy style, and reStructuredText (Sphinx). The specific style matters less than consistency — pick one and use it throughout a project.

Variable Scope Inside Functions

Every variable defined inside a function exists only within that function. This is called local scope. When the function finishes, its local variables are discarded. This is a feature, not a limitation: it means that intermediate variables created during data processing — counters, temporary lists, loop variables — do not leak into the rest of the script and cannot accidentally overwrite something important.

total = 1000   # global variable

def compute_tax(amount, rate=0.08):
    tax = amount * rate      # 'tax' is local to this function
    net = amount + tax       # 'net' is also local
    return net

result = compute_tax(500)
print(result)   # 540.0
print(tax)      # NameError: name 'tax' is not defined

The variable total defined outside the function is in the global scope. A function can read a global variable without any special declaration. However, to modify a global variable from inside a function, you must explicitly declare it with the global keyword:

record_count = 0

def process_record(record):
    global record_count
    record_count += 1
    # ... processing logic ...

While the global keyword works, using it is generally discouraged. Functions that modify global state are harder to test, harder to reason about, and can introduce subtle bugs — especially in larger scripts where many functions might interact with the same global variable. The preferred pattern is to pass data in as arguments and return results explicitly, keeping each function self-contained:

# Preferred: no global modification
def process_all_records(records):
    count = 0
    for record in records:
        # ... process each record ...
        count += 1
    return count

total_processed = process_all_records(my_records)

Understanding scope also prevents a common class of bug where a developer expects a function to have modified some data structure, but because Python sometimes works with copies rather than references (depending on the type), the original remains unchanged — or vice versa, a function unintentionally mutates a list that was supposed to stay intact elsewhere in the script. The safest habit is to treat function parameters as read-only inside the function body, working on copies when modification is needed:

def cap_outliers(data, upper_limit):
    """Return a new list with values capped at upper_limit."""
    return [min(x, upper_limit) for x in data]   # original 'data' untouched

The following table summarises the key concepts covered in this topic and their practical relevance in data analytics work:

Concept Syntax / Keyword Primary Benefit in Data Analytics
Function definition def name(): Encapsulates reusable processing logic
Parameters def f(a, b): Makes functions flexible across different datasets
Default parameters def f(a, b=2): Provides sensible defaults while allowing overrides
Keyword arguments f(a=1, b=2) Improves readability at the call site
Return values return value Passes transformed data to the next pipeline step
Multiple returns return a, b, c Returns several statistics in a single call
Docstrings """...""" Documents intent for team use and tool support
Local scope Variables inside def Prevents temporary variables from polluting global state
Global keyword global var Allowed but discouraged; prefer return values

Putting all these ideas together, a well-structured analytics script defines small, clearly named, documented functions near the top; passes data explicitly through parameters; captures results with meaningful variable names; and reserves the main execution flow for calling those functions in sequence. This approach makes scripts easier to read, debug, test, and extend — whether you are working alone or as part of a team.
