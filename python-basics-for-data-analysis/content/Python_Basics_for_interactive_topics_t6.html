Writing Foundational Data Analysis Scripts

▶

Data analysis with Python is not simply a matter of knowing individual language features in isolation — it is about combining those features into a coherent, repeatable workflow that takes raw data from a file and turns it into meaningful insight. A well-written data analysis script reads almost like a story: data enters, it is cleaned and shaped, calculations are performed, and findings are presented clearly. This topic brings together Python fundamentals — file I/O, control structures, functions, data types, and output formatting — and shows how they fit together into complete, professional-quality scripts. Whether you are summarizing a spreadsheet of sales figures or exploring a dataset of survey responses, the same structural principles apply.

Structuring a Complete Data Analysis Script

The single most important habit you can build as a data analyst is giving your scripts a clear, predictable structure. A script that is organized into named sections is far easier to debug, hand off to a colleague, or return to after a week away. Think of the structure as a pipeline: each section receives the outputs of the previous section, transforms or uses them, and passes its own outputs forward.

A conventional layout follows four broad stages:

  • Imports — Every script begins with import statements. Placing all imports at the top makes dependencies immediately visible and avoids the confusion of discovering a missing library halfway through execution.
  • Load data — Read the raw data from its source (a file, a database query, an API) and store it in an in-memory structure.
  • Clean and transform — Correct types, remove invalid records, derive new fields, and reshape the data into a form suited to analysis.
  • Analyze and report — Compute summary statistics, apply business logic, and produce output — whether printed to the console or written to a file.

Section-dividing comments such as # --- Load Data --- or # === Summary Statistics === are a lightweight but powerful documentation tool. They cost nothing and make the script's intent legible at a glance. A reader skimming the file can jump straight to the section they care about without reading every line. A clear structure also enforces a healthy dependency discipline: the analysis section can only use data that the loading and cleaning sections have already prepared, which prevents a common class of bugs where a variable is used before it has been properly initialized or validated.

import csv
import os

# --- Configuration ---
INPUT_FILE  = "sales_data.csv"
OUTPUT_FILE = "sales_summary.txt"

# --- Load Data ---
# ... (loading logic here)

# --- Clean and Transform ---
# ... (type conversion, filtering here)

# --- Analyze ---
# ... (summary statistics here)

# --- Report Results ---
# ... (output logic here)

Even before a single line of real logic is written, this skeleton communicates the script's shape and intent.

Reading Datasets from Files

Most real-world datasets live in files — commonly plain text or CSV (comma-separated values) format. Python gives you two complementary tools: the built-in open() function for raw file access, and the standard library's csv module for structured CSV parsing.

The open() function returns a file object. Using it as a context manager with the with statement is best practice because Python automatically closes the file when the block exits, even if an exception occurs:

with open("sales_data.csv", "r", newline="") as f:
    content = f.read()          # read entire file as one string

For CSV files specifically, the csv module handles the details of quoted fields, embedded commas, and line endings that would be tedious to handle manually. csv.DictReader is especially useful because it uses the first row as field names and returns each subsequent row as a dictionary, making the data self-documenting:

import csv

records = []
with open("sales_data.csv", "r", newline="") as f:
    reader = csv.DictReader(f)
    for row in reader:
        records.append(row)

# records is now a list of dicts, e.g.:
# [{'date': '2024-01-03', 'product': 'Widget', 'units': '14', 'revenue': '210.00'}, ...]

A critical robustness concern is handling files that do not exist. Any production-quality script must account for this with a try/except block. Without it, a missing file causes an unhandled FileNotFoundError that crashes the script and gives the user no helpful guidance:

records = []
try:
    with open("sales_data.csv", "r", newline="") as f:
        reader = csv.DictReader(f)
        for row in reader:
            records.append(row)
except FileNotFoundError:
    print("Error: 'sales_data.csv' was not found. Please check the file path.")
    raise SystemExit(1)

Storing loaded data as a list of dictionaries is a natural and flexible choice. The list preserves row order, supports iteration, and can be sliced or filtered. Each dictionary maps column names to values, which makes downstream code readable — row["revenue"] is far clearer than row[3].

Applying Control Structures to Process Data

Once data is in memory, control structures — loops and conditionals — are the workhorses that process it. A for loop lets you visit every record exactly once and apply the same transformation or check uniformly across the entire dataset:

# Tag each record with its sales tier
for record in records:
    units = int(record["units"])
    if units >= 50:
        record["tier"] = "High"
    elif units >= 20:
        record["tier"] = "Medium"
    else:
        record["tier"] = "Low"

Conditional statements (if / elif / else) let you branch logic based on data values. Common use cases include filtering out invalid rows, assigning categorical labels (as shown above), flagging outliers, or routing records into different accumulators for group-level analysis.

Nested loops and conditions handle multi-level logic. For example, grouping records by product category before summing revenue requires an outer structure (a dictionary keyed by category) populated inside the loop:

revenue_by_product = {}

for record in records:
    product = record["product"]
    revenue = float(record["revenue"])

    if product not in revenue_by_product:
        revenue_by_product[product] = 0.0
    revenue_by_product[product] += revenue

A practical discipline here is keeping each loop body focused on one responsibility. If a loop is simultaneously filtering, transforming, accumulating, and printing, it becomes difficult to test or reason about. When a loop body grows long, that is usually a signal to extract part of it into a helper function.

Computing Summary Statistics with Functions

Functions are the mechanism for turning repeated patterns of logic into named, reusable units. For data analysis, the most common summary statistics are total (sum), average (mean), minimum, and maximum. Defining functions for these keeps the analysis section of the script clean and makes it trivial to apply the same calculation to different columns.

def compute_total(values):
    """Return the sum of a list of numeric values."""
    return sum(values)

def compute_average(values):
    """Return the mean of a list of numeric values, or 0 if the list is empty."""
    if len(values) == 0:
        return 0
    return sum(values) / len(values)

def compute_min(values):
    return min(values)

def compute_max(values):
    return max(values)

Each function accepts a list as its argument, which means you can call it on any column simply by extracting that column's values first:

revenues = [float(r["revenue"]) for r in records]
units    = [int(r["units"])    for r in records]

total_revenue   = compute_total(revenues)
average_revenue = compute_average(revenues)
peak_units      = compute_max(units)

Notice that the functions return their results rather than printing them. This is a crucial design choice: a function that prints directly is hard to reuse because you cannot capture its output for further computation. By returning a value, the caller decides what to do with it — store it, pass it to another function, or format it for output.

Python's built-in sum(), len(), min(), and max() already handle simple cases efficiently. Custom functions add value when you need guarded behavior (like the empty-list check in compute_average), weighted calculations, or domain-specific logic (such as computing a median or a percentage contribution).

Integrating Data Types Appropriately

A near-universal source of bugs in file-based data analysis is forgetting that everything read from a file is a string. When csv.DictReader gives you row["units"] == "14", that is the string "14", not the integer 14. Adding two such strings gives "1420" (concatenation), not 34 (addition). Type conversion must happen explicitly.

The cleaning stage is the right place to perform these conversions, so that all downstream code can assume correct types:

clean_records = []

for row in records:
    try:
        clean_records.append({
            "date":    row["date"],
            "product": row["product"],
            "units":   int(row["units"]),
            "revenue": float(row["revenue"]),
        })
    except (ValueError, KeyError) as e:
        print(f"Skipping malformed row {row}: {e}")

Wrapping conversions in try/except catches rows where a field is missing or contains a non-numeric value (such as an empty string or the word "N/A"). Rather than crashing, the script skips the bad row and logs a warning — a far better experience for the user.

Choosing the right data structure matters too. The table below summarizes when to reach for each core Python container in a data analysis context:

Structure Best Used For Example in Analysis
list Ordered sequences of records or extracted values All rows from a CSV; a column of revenue figures
dict Named fields for a single record; grouping accumulators One CSV row; revenue totals keyed by product
set Unique values; membership testing Distinct product names in the dataset
tuple Fixed, immutable groupings A (min, max) pair returned from a function

isinstance() can guard logic that depends on type when you are not certain what a variable holds:

def safe_float(value):
    if isinstance(value, float):
        return value
    try:
        return float(value)
    except (ValueError, TypeError):
        return 0.0

Outputting and Presenting Results

Computed results have no value if they are not communicated clearly. Python's f-strings (formatted string literals, introduced in Python 3.6) make it easy to embed variables directly in text and control numeric formatting:

print(f"Total Revenue:    ${total_revenue:,.2f}")
print(f"Average Revenue:  ${average_revenue:,.2f}")
print(f"Peak Units Sold:  {peak_units}")

The format specification :,.2f means: use a comma as the thousands separator and show exactly two decimal places. This kind of formatting is essential when presenting financial figures. Without it, 210000.5 appears instead of $210,000.50 — technically correct but hard to read.

When results need to be saved or shared, write them to a file instead of (or in addition to) printing:

with open("sales_summary.txt", "w") as out:
    out.write("=== Sales Summary ===\n\n")
    out.write(f"Total Revenue:   ${total_revenue:,.2f}\n")
    out.write(f"Average Revenue: ${average_revenue:,.2f}\n\n")
    out.write("--- Revenue by Product ---\n")
    for product, rev in sorted(revenue_by_product.items()):
        out.write(f"  {product:<20} ${rev:,.2f}\n")

Organizing output logically makes results easier to interpret. A common convention is to present high-level totals first, then drill into category-level breakdowns. This mirrors the way a reader naturally wants to consume information — understand the whole before examining the parts. Labeling every value explicitly ("Total Revenue", not just the number) ensures the output file is self-explanatory without requiring someone to read the script to understand what the numbers mean.

Bringing It All Together: An End-to-End Script Workflow

The following complete script demonstrates every principle discussed above, applied to a realistic scenario: reading a CSV file of sales transactions, cleaning and typing the data, computing overall and per-product summary statistics, and writing a formatted report.

import csv
import os

# --- Configuration ---
INPUT_FILE  = "sales_data.csv"
OUTPUT_FILE = "sales_summary.txt"

# --- Helper Functions ---

def load_records(filepath):
    """Load CSV file and return a list of raw string dicts."""
    rows = []
    try:
        with open(filepath, "r", newline="") as f:
            reader = csv.DictReader(f)
            for row in reader:
                rows.append(row)
    except FileNotFoundError:
        print(f"Error: '{filepath}' not found.")
        raise SystemExit(1)
    return rows

def clean_records(raw_rows):
    """Convert string fields to appropriate types; skip malformed rows."""
    clean = []
    for row in raw_rows:
        try:
            clean.append({
                "date":    row["date"],
                "product": row["product"].strip(),
                "units":   int(row["units"]),
                "revenue": float(row["revenue"]),
            })
        except (ValueError, KeyError) as e:
            print(f"Warning: skipping row {row} — {e}")
    return clean

def compute_average(values):
    """Return mean of a non-empty list, else 0."""
    return sum(values) / len(values) if values else 0

def group_by_product(records):
    """Return a dict mapping product name to total revenue."""
    totals = {}
    for r in records:
        product = r["product"]
        totals[product] = totals.get(product, 0.0) + r["revenue"]
    return totals

def write_report(filepath, records, product_totals):
    """Write a formatted summary report to a text file."""
    revenues = [r["revenue"] for r in records]
    units    = [r["units"]   for r in records]

    with open(filepath, "w") as out:
        out.write("========================================\n")
        out.write("           SALES ANALYSIS REPORT\n")
        out.write("========================================\n\n")

        out.write("OVERALL SUMMARY\n")
        out.write(f"  Records Processed : {len(records)}\n")
        out.write(f"  Total Revenue     : ${sum(revenues):,.2f}\n")
        out.write(f"  Average Revenue   : ${compute_average(revenues):,.2f}\n")
        out.write(f"  Total Units Sold  : {sum(units)}\n")
        out.write(f"  Min Units (sale)  : {min(units)}\n")
        out.write(f"  Max Units (sale)  : {max(units)}\n\n")

        out.write("REVENUE BY PRODUCT\n")
        for product, rev in sorted(product_totals.items(),
                                   key=lambda x: x[1], reverse=True):
            out.write(f"  {product:<25} ${rev:,.2f}\n")

        out.write("\n========================================\n")
    print(f"Report written to '{filepath}'.")

# --- Main Workflow ---

raw_rows      = load_records(INPUT_FILE)
records       = clean_records(raw_rows)
product_totals = group_by_product(records)
write_report(OUTPUT_FILE, records, product_totals)

Several practices in this script are worth highlighting explicitly:

  • Incremental development: Build and test the script in stages. First, call load_records() and print a few rows to confirm the file is being read correctly. Then add clean_records() and verify types. Then add group_by_product() and print the dictionary. Finally, add write_report(). Each stage gives you a checkpoint where you can confirm correctness before adding complexity.
  • Function reuse: compute_average() is defined once and used inside write_report(). If the averaging logic ever needs to change — say, to compute a weighted average — there is only one place to update.
  • Separation of concerns: Loading, cleaning, analysis, and output are handled by separate functions. None of them does another's job. This makes each function independently testable and the main workflow easy to read.
  • Self-documenting output: The report file labels every figure and uses consistent alignment. A manager reading the output file needs no knowledge of the script to understand what the numbers represent.
  • Robustness: File-not-found is handled gracefully; malformed rows are skipped with a warning rather than crashing the script.

Planning before coding is equally important. Before writing the first line, sketch out the stages, identify what inputs each stage needs, and decide what structure will hold the data between stages. A five-minute planning sketch prevents hours of restructuring later. Once the script is complete, read it top to bottom as if you were seeing it for the first time — if any section's purpose is unclear, add a comment or rename a variable. Code that is readable is code that can be trusted, maintained, and extended.
