Merging, Joining, and Concatenating Data

▶

Combining data from multiple sources is one of the most frequent and consequential tasks in any data analysis workflow. Real-world data rarely arrives pre-assembled into a single, tidy table. Instead, you might have customer records in one file, transaction histories in another, and product details in a third. Pandas provides a rich toolkit for unifying these scattered datasets: pd.concat() for stacking DataFrames together, pd.merge() for SQL-style relational joins on key columns, and the .join() method for index-based joining. Understanding when and how to use each tool — and how to verify that the resulting combined dataset is correct — is essential for trustworthy data analysis.

Concatenating DataFrames with pd.concat()

Concatenation is the simplest form of combination: you are physically gluing DataFrames together either vertically (stacking rows) or horizontally (placing columns side by side). The pd.concat() function accepts a list of DataFrames and a handful of parameters that control exactly how the gluing works.

Vertical concatenation with axis=0 appends the rows of one DataFrame below another. This is conceptually the same as running a SQL UNION ALL. It is the default behavior of pd.concat().

import pandas as pd

df1 = pd.DataFrame({
    'employee_id': [101, 102],
    'name': ['Alice', 'Bob'],
    'department': ['Engineering', 'Marketing']
})

df2 = pd.DataFrame({
    'employee_id': [103, 104],
    'name': ['Carol', 'David'],
    'department': ['Finance', 'Engineering']
})

result = pd.concat([df1, df2], axis=0)
print(result)
#    employee_id   name   department
# 0          101  Alice  Engineering
# 1          102    Bob    Marketing
# 0          103  Carol      Finance
# 1          104  David  Engineering

Notice that the index values from both original DataFrames are preserved, which means the resulting DataFrame can contain duplicate index labels (0 and 1 appear twice). To reset the index to a clean 0-based sequence, pass ignore_index=True.

Horizontal concatenation with axis=1 places DataFrames side by side, aligning rows by their index values. This is useful when you have two DataFrames that describe different attributes of the same set of observations and share the same index.

scores = pd.DataFrame({
    'math_score': [88, 92, 79]
}, index=['Alice', 'Bob', 'Carol'])

grades = pd.DataFrame({
    'letter_grade': ['B', 'A', 'C']
}, index=['Alice', 'Bob', 'Carol'])

combined = pd.concat([scores, grades], axis=1)
print(combined)
#        math_score letter_grade
# Alice          88            B
# Bob            92            A
# Carol          79            C

The keys parameter creates a hierarchical (MultiIndex) index in the result, which labels which source DataFrame each row came from. This is invaluable when you need to trace data provenance after concatenation.

result = pd.concat([df1, df2], axis=0, keys=['batch_1', 'batch_2'])
print(result)
#                  employee_id   name   department
# batch_1 0               101  Alice  Engineering
#         1               102    Bob    Marketing
# batch_2 0               103  Carol      Finance
#         1               104  David  Engineering

You can later use result.loc['batch_1'] to retrieve only the rows from the first source.

Mismatched columns during vertical concatenation produce NaN values for the missing entries. If df1 has a column that df2 does not, every row contributed by df2 will have NaN in that column, and vice versa. This behavior mirrors a SQL UNION ALL with differing column sets.

df_a = pd.DataFrame({'id': [1, 2], 'color': ['red', 'blue']})
df_b = pd.DataFrame({'id': [3, 4], 'size': ['small', 'large']})

result = pd.concat([df_a, df_b], axis=0, ignore_index=True)
print(result)
#    id color   size
# 0   1   red    NaN
# 1   2  blue    NaN
# 2   3   NaN  small
# 3   4   NaN  large

To suppress this and keep only shared columns, pass join='inner' to pd.concat().

Merging DataFrames with pd.merge()

While concatenation simply glues DataFrames together structurally, merging performs a relational join — it matches rows from two DataFrames based on shared key values. This is conceptually identical to SQL JOIN operations and is the right tool when rows need to be aligned by the content of one or more columns, not by their position or index.

The on parameter specifies the column name (or list of column names) that act as the join key. Both DataFrames must contain a column with exactly that name.

orders = pd.DataFrame({
    'order_id': [1, 2, 3],
    'customer_id': [201, 202, 201],
    'amount': [150.0, 200.0, 75.0]
})

customers = pd.DataFrame({
    'customer_id': [201, 202, 203],
    'name': ['Alice', 'Bob', 'Carol'],
    'city': ['New York', 'London', 'Paris']
})

merged = pd.merge(orders, customers, on='customer_id')
print(merged)
#    order_id  customer_id  amount   name      city
# 0         1          201   150.0  Alice  New York
# 1         3          201    75.0  Alice  New York
# 2         2          202   200.0    Bob    London

The left_on and right_on parameters handle the common real-world situation where the same conceptual key has different column names in each DataFrame. For example, one table might call it customer_id while the other calls it cust_no.

invoices = pd.DataFrame({
    'invoice_id': [1001, 1002],
    'cust_no': [201, 202],
    'total': [300.0, 450.0]
})

merged = pd.merge(invoices, customers, left_on='cust_no', right_on='customer_id')
print(merged)
#    invoice_id  cust_no  total  customer_id   name      city
# 0        1001      201  300.0          201  Alice  New York
# 1        1002      202  450.0          202    Bob    London

Note that both the cust_no and customer_id columns appear in the result because they were treated as separate (though equal-valued) columns. You may wish to drop one afterwards.

The how parameter controls which rows survive the join. The four options — 'inner', 'outer', 'left', and 'right' — correspond directly to standard SQL join semantics and are explained in depth in the next section.

Types of Joins: Inner, Outer, Left, and Right

The choice of join type determines the shape and completeness of the merged result. Each type makes a different decision about what to do with keys that exist in one DataFrame but not the other.

Inner join (how='inner') — the default — returns only the rows whose key value appears in both DataFrames. It is the intersection of keys. Rows from either side that have no match in the other are silently dropped.

left = pd.DataFrame({'key': ['A', 'B', 'C'], 'val_left': [1, 2, 3]})
right = pd.DataFrame({'key': ['B', 'C', 'D'], 'val_right': [10, 20, 30]})

inner = pd.merge(left, right, on='key', how='inner')
print(inner)
#   key  val_left  val_right
# 0   B         2         10
# 1   C         3         20

Key 'A' exists only in left and key 'D' exists only in right; both are excluded from the result.

Outer join (how='outer') returns all rows from both DataFrames — the union of keys. Where a key is missing on one side, the corresponding columns are filled with NaN.

outer = pd.merge(left, right, on='key', how='outer')
print(outer)
#   key  val_left  val_right
# 0   A       1.0        NaN
# 1   B       2.0       10.0
# 2   C       3.0       20.0
# 3   D       NaN       30.0

Left join (how='left') keeps every row from the left DataFrame and brings in matching data from the right where possible. Unmatched right rows are discarded; unmatched left rows receive NaN in the right DataFrame's columns.

left_join = pd.merge(left, right, on='key', how='left')
print(left_join)
#   key  val_left  val_right
# 0   A         1        NaN
# 1   B         2       10.0
# 2   C         3       20.0

This is the most common join type in practice because it preserves the completeness of a "primary" dataset while enriching it with optional supplementary information.

Right join (how='right') is the mirror image: it keeps every row from the right DataFrame and fills from the left where possible.

right_join = pd.merge(left, right, on='key', how='right')
print(right_join)
#   key  val_left  val_right
# 0   B       2.0         10
# 1   C       3.0         20
# 2   D       NaN         30

The following table summarizes which rows are retained under each join type:

Join Type Rows from Left Rows from Right Unmatched Handling
inner Only matched Only matched Unmatched rows dropped from both sides
outer All All NaN filled for missing columns on either side
left All Only matched NaN filled in right columns for unmatched left rows
right Only matched All NaN filled in left columns for unmatched right rows

Joining DataFrames Using the .join() Method

The .join() method is a convenience wrapper around pd.merge() that is specifically optimized for index-based joining. It is called on a DataFrame instance and joins it against another DataFrame (or list of DataFrames) by default using their indices.

df_info = pd.DataFrame(
    {'department': ['Engineering', 'Marketing', 'Finance']},
    index=['Alice', 'Bob', 'Carol']
)

df_salary = pd.DataFrame(
    {'salary': [95000, 72000, 85000]},
    index=['Alice', 'Bob', 'Carol']
)

result = df_info.join(df_salary)
print(result)
#        department  salary
# Alice  Engineering   95000
# Bob      Marketing   72000
# Carol      Finance   85000

The default behavior is a left join: all rows from the calling DataFrame are preserved, and only matching index values from the other DataFrame are brought in. The how parameter works the same as in pd.merge() and can be set to 'inner', 'outer', or 'right'.

The on parameter in .join() changes the behavior: instead of joining on the calling DataFrame's index, it joins on a specified column of the calling DataFrame against the other DataFrame's index. This bridges the gap between index-based and column-based joining.

employees = pd.DataFrame({
    'name': ['Alice', 'Bob', 'Carol'],
    'dept_code': ['ENG', 'MKT', 'FIN']
})

dept_info = pd.DataFrame(
    {'full_name': ['Engineering', 'Finance', 'Marketing']},
    index=['ENG', 'FIN', 'MKT']
)

result = employees.join(dept_info, on='dept_code')
print(result)
#     name dept_code    full_name
# 0  Alice       ENG  Engineering
# 1    Bob       MKT    Marketing
# 2  Carol       FIN      Finance

The lsuffix and rsuffix parameters resolve naming conflicts when both DataFrames share non-key column names. Without specifying them, Pandas raises a ValueError if there are overlapping column names.

df_x = pd.DataFrame({'score': [80, 90]}, index=['Alice', 'Bob'])
df_y = pd.DataFrame({'score': [75, 95]}, index=['Alice', 'Bob'])

result = df_x.join(df_y, lsuffix='_exam1', rsuffix='_exam2')
print(result)
#        score_exam1  score_exam2
# Alice           80           75
# Bob             90           95

Joining multiple DataFrames at once is possible by passing a list to .join(). Pandas performs sequential left joins across all DataFrames in the list, making it a compact way to combine many index-aligned tables.

df_a = pd.DataFrame({'col_a': [1, 2]}, index=['r1', 'r2'])
df_b = pd.DataFrame({'col_b': [3, 4]}, index=['r1', 'r2'])
df_c = pd.DataFrame({'col_c': [5, 6]}, index=['r1', 'r2'])

result = df_a.join([df_b, df_c])
print(result)
#     col_a  col_b  col_c
# r1      1      3      5
# r2      2      4      6

Merging on Multiple Keys

A single key column is often not sufficient to uniquely identify matching rows between two DataFrames. For example, a sales table and a quota table might both require matching on both the salesperson ID and the year to find the correct corresponding row. In such cases, you define a composite key by passing a list of column names to the on parameter.

sales = pd.DataFrame({
    'rep_id': [1, 1, 2, 2],
    'year':   [2022, 2023, 2022, 2023],
    'revenue': [120000, 135000, 98000, 110000]
})

quotas = pd.DataFrame({
    'rep_id': [1, 1, 2, 2],
    'year':   [2022, 2023, 2022, 2023],
    'quota':  [100000, 130000, 95000, 105000]
})

merged = pd.merge(sales, quotas, on=['rep_id', 'year'])
print(merged)
#    rep_id  year  revenue   quota
# 0       1  2022   120000  100000
# 1       1  2023   135000  130000
# 2       2  2022    98000   95000
# 3       2  2023   110000  105000

Without the composite key, a merge on rep_id alone would produce a cartesian-product-like result where each rep's 2022 row is matched with both their 2022 and 2023 quota, doubling the rows erroneously.

All key columns must appear in both DataFrames when using on. If the column names differ between the two DataFrames, use left_on and right_on with lists of column names in matching positional order.

sales_renamed = sales.rename(columns={'rep_id': 'salesperson', 'year': 'fiscal_year'})

merged = pd.merge(
    sales_renamed, quotas,
    left_on=['salesperson', 'fiscal_year'],
    right_on=['rep_id', 'year']
)

Every key column must match simultaneously for a row pair to be included in an inner merge. A row from the left with rep_id=1, year=2022 will only match a row from the right that also has exactly rep_id=1 AND year=2022. Partial matches on a subset of composite keys are not treated as matches.

Handling Duplicate Columns and Suffixes After Merging

When two DataFrames share column names that are not part of the join key, those columns will both appear in the merged result. By default, Pandas disambiguates them by appending _x to the column from the left DataFrame and _y to the column from the right DataFrame.

products = pd.DataFrame({
    'product_id': [10, 20],
    'name': ['Widget', 'Gadget'],
    'price': [9.99, 24.99]
})

reviews = pd.DataFrame({
    'product_id': [10, 20],
    'name': ['Widget', 'Gadget'],
    'rating': [4.5, 3.8]
})

merged = pd.merge(products, reviews, on='product_id')
print(merged.columns.tolist())
# ['product_id', 'name_x', 'price', 'name_y', 'rating']

The generic _x and _y suffixes can be confusing in complex pipelines. The suffixes parameter lets you assign meaningful labels:

merged = pd.merge(products, reviews, on='product_id', suffixes=('_product', '_review'))
print(merged.columns.tolist())
# ['product_id', 'name_product', 'price', 'name_review', 'rating']

After merging, it is good practice to inspect and clean up duplicate columns. You might verify that the duplicated columns contain identical values (implying they are truly redundant) and then drop one:

# Verify they are identical
assert (merged['name_product'] == merged['name_review']).all()

# Drop the redundant column
merged = merged.drop(columns=['name_review'])
merged = merged.rename(columns={'name_product': 'name'})

Failing to handle duplicate columns creates downstream hazards: aggregations may operate on the wrong column, filters may silently reference an ambiguous name, and exported reports become confusing for stakeholders.

Validating and Verifying Merge Results

A merge operation can silently produce a result that looks plausible but is subtly wrong — for example, row counts may balloon due to one-to-many relationships, or shrink unexpectedly due to unmatched keys. Systematic verification is therefore not optional; it is a professional discipline.

Check the shape of the merged DataFrame. Before and after the merge, reason about what the row count should be. In a one-to-one inner join between tables of length n and m, the result should have at most min(n, m) rows. A row count larger than either input table is a strong signal of an unintended one-to-many or many-to-many join.

print(f"Left rows: {len(orders)}, Right rows: {len(customers)}, Merged rows: {len(merged)}")

The validate parameter enforces the expected relationship cardinality and raises a MergeError if it is violated. The accepted values are:

validate value Enforced relationship
'1:1' Each key appears at most once in both DataFrames
'1:m' Each key appears at most once in the left DataFrame
'm:1' Each key appears at most once in the right DataFrame
'm:m' No uniqueness constraint enforced (default behavior) 
# This will raise MergeError if customer_id is not unique in customers
merged = pd.merge(orders, customers, on='customer_id', validate='m:1')

Inspect NaN counts in the merged columns to identify keys that failed to match. This is particularly useful after a left or outer join, where unmatched rows are retained but filled with NaN.

nan_counts = merged.isnull().sum()
print(nan_counts)
# Columns with high NaN counts indicate poor join key overlap

The indicator=True parameter adds a special column called _merge to the result. Each row is labeled with one of three string values: 'left_only' (the key existed only in the left DataFrame), 'right_only' (only in the right), or 'both' (matched in both). This makes it trivial to audit which rows matched and which did not.

merged = pd.merge(left, right, on='key', how='outer', indicator=True)
print(merged)
#   key  val_left  val_right      _merge
# 0   A       1.0        NaN   left_only
# 1   B       2.0       10.0        both
# 2   C       3.0       20.0        both
# 3   D       NaN       30.0  right_only

# Count by category
print(merged['_merge'].value_counts())
# both          2
# left_only     1
# right_only    1

You can filter the _merge column to extract the unmatched rows for investigation — for instance, to find orders referencing a customer_id that does not exist in the customer master table, which would indicate a data quality problem. Once the indicator column has served its diagnostic purpose, drop it with merged.drop(columns=['_merge']) before proceeding with analysis.
