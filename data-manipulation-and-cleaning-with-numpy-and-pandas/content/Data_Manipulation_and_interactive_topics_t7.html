Cleaning and Transforming Real-World Datasets

▶

Real-world datasets are almost never ready for analysis straight out of the source. They arrive with missing values scattered across dozens of columns, dates stored as plain strings, duplicate records introduced by ETL bugs, outliers that distort statistical summaries, and free-text fields that encode multiple pieces of information in a single messy column. Professional data scientists spend a substantial portion of their time—often cited at 60–80% of a project—on exactly this kind of preparation work. The goal of a thorough cleaning and transformation workflow is not just to fix obvious errors; it is to produce a dataset whose quality can be demonstrated, whose provenance can be traced, and whose preparation logic can be reapplied automatically to every future batch of data that arrives from the same source. This topic walks through every major stage of that workflow in depth, from the first diagnostic look at a raw file all the way through to validated export.

Establishing an End-to-End Cleaning Workflow

Before touching a single value, the first responsibility is to understand what you actually have. Three Pandas methods together give a rapid, comprehensive portrait of any DataFrame. df.info() prints the number of rows, the name and dtype of every column, and the count of non-null entries per column, making it immediately obvious where data is absent and where types are wrong. df.describe() computes count, mean, standard deviation, minimum, quartiles, and maximum for every numeric column, revealing scale mismatches, impossibly large values, and columns where the mean and median diverge sharply. df.head() (and df.tail()) show literal cell values, which often exposes formatting artifacts—leading spaces, inconsistent capitalisation, currency symbols embedded in numeric fields—that statistics alone cannot reveal.

import pandas as pd
import numpy as np

df = pd.read_csv('sales_raw.csv')

print(df.info())        # shape, dtypes, non-null counts
print(df.describe())    # numeric distributions
print(df.head(10))      # sample rows

Running these three lines at the start of every project is a non-negotiable habit. They form the audit that drives every subsequent decision. Once the audit is complete, the cleaning steps should be assembled into a repeatable pipeline—either a function, a class, or an ordered series of cells in a notebook that is treated as code rather than a scratchpad. The key design principle is idempotency: running the pipeline twice on the same raw data should produce exactly the same cleaned output. This matters because data sources are rarely static; new monthly exports will need to go through the same transformations.

At every meaningful stage of the pipeline—after type conversion, after imputation, after deduplication—it is good practice to insert assertions that act as automated checks. If an assertion fails, the pipeline stops with a clear error rather than silently passing corrupt data downstream.

def validate_after_type_conversion(df):
    assert df['order_date'].dtype == 'datetime64[ns]', "order_date must be datetime"
    assert df['revenue'].dtype in ['float64', 'int64'], "revenue must be numeric"
    assert df.shape[1] == 12, f"Expected 12 columns, got {df.shape[1]}"
    print("Type conversion validation passed.")

validate_after_type_conversion(df)

This style of defensive programming transforms the pipeline from a fragile sequence of manual steps into a self-documenting, auditable process that raises a hand the moment something unexpected happens.

Diagnosing and Handling Missing Data at Scale

Missing data is the most pervasive quality problem in real datasets. The first step is always to quantify it, not guess at it. df.isnull().mean() returns the fraction of missing values per column as a float between 0 and 1, making it straightforward to apply a threshold-based decision rule.

missing_fractions = df.isnull().mean().sort_values(ascending=False)
print(missing_fractions)

# Drop columns where more than 40% of values are missing
threshold = 0.40
df = df.loc[:, missing_fractions <= threshold]

Columns missing more than 40% of their values rarely contribute reliable signal to analysis or models; retaining them risks introducing more noise than information. For columns below the threshold, the appropriate imputation strategy depends on the nature of the data and the mechanism behind the missingness.

Mean, median, and mode imputation with fillna() is appropriate when data is missing at random—that is, the fact that a value is missing is unrelated to the value itself. Mean imputation works for roughly symmetric numeric distributions; median imputation is more robust when the distribution is skewed because the median is not pulled by extreme values; mode imputation is the standard choice for categorical columns.

# Numeric: fill with median (robust to outliers)
df['age'] = df['age'].fillna(df['age'].median())

# Categorical: fill with most frequent value
df['region'] = df['region'].fillna(df['region'].mode()[0])

Forward-fill and backward-fill are the correct strategy for time-series or any dataset where rows represent sequential observations and the nearest known value is the best available estimate. Forward-fill (ffill) propagates the last valid observation forward; backward-fill (bfill) propagates the next valid observation backward. Both are available as arguments to fillna() or as dedicated methods.

# Time-ordered sensor readings: carry last known value forward
df['temperature'] = df['temperature'].fillna(method='ffill')

# Or equivalently:
df['temperature'] = df['temperature'].ffill()

For datasets with complex, non-random missingness patterns, interpolation via df.interpolate() estimates missing values from the surrounding data using a chosen mathematical method. Linear interpolation (the default) fits a straight line between the nearest non-null neighbours; polynomial and spline methods are available for smoother estimation curves.

# Linear interpolation between known values
df['stock_price'] = df['stock_price'].interpolate(method='linear')

When missingness is structural and correlated with other features—for example, income is missing specifically for respondents who skipped the survey question—simple statistical imputation introduces bias. In those cases, model-based imputation using scikit-learn's IterativeImputer (which models each feature with missing values as a function of the others) or KNNImputer (which fills values from the k nearest complete rows) produces much more defensible estimates. These estimators accept and return NumPy arrays, so the workflow is to extract the relevant columns, impute, and write back.

from sklearn.impute import KNNImputer

imputer = KNNImputer(n_neighbors=5)
cols_to_impute = ['income', 'savings', 'debt']
df[cols_to_impute] = imputer.fit_transform(df[cols_to_impute])

Correcting and Converting Data Types

A freshly loaded CSV assigns every column one of only a handful of inferred dtypes—usually object, int64, or float64. The inferred type is frequently wrong. A column of dates arrives as object. A column of integer IDs arrives as float64 because one row had a missing value. A column of category labels arrives as object, consuming far more memory than necessary. Correcting these types early in the pipeline improves both correctness and performance for everything that follows.

The general-purpose tool is df['col'].astype(). It accepts Python built-in types, NumPy dtype strings, and Pandas extension types.

df['customer_id'] = df['customer_id'].astype('int64')
df['score']       = df['score'].astype('float64')
df['is_active']   = df['is_active'].astype('bool')

Parsing dates requires special treatment. pd.to_datetime() attempts automatic inference of the format, but automatic inference is slow on large datasets and can silently misparse ambiguous dates (is 01/02/03 January 2nd or February 1st?). Providing the format parameter removes ambiguity and is significantly faster.

df['order_date'] = pd.to_datetime(df['order_date'], format='%Y-%m-%d')

# If the source mixes formats, use errors='coerce' to turn failures into NaT
df['order_date'] = pd.to_datetime(df['order_date'], errors='coerce')

Once a column is datetime64, Pandas exposes the full .dt accessor, enabling clean extraction of year, month, day, weekday, hour, and so on—all without string parsing.

df['order_year']  = df['order_date'].dt.year
df['order_month'] = df['order_date'].dt.month
df['day_of_week'] = df['order_date'].dt.day_name()

The category dtype is one of the most underused optimisations in Pandas. A column of string labels with low cardinality—country names, product categories, status codes—stored as object holds a full Python string object for every single row. Converting to category stores each unique label once in a lookup table and replaces each cell with a small integer index. For a million-row DataFrame with a column of 50 country names, the memory reduction can be 10× or more, and groupby operations on category columns run considerably faster.

df['country']  = df['country'].astype('category')
df['status']   = df['status'].astype('category')

print(df.memory_usage(deep=True))   # compare before and after

When a column should be numeric but contains textual noise—cells like "N/A", "--", or "12.4 USD"—a direct astype(float) raises an exception. pd.to_numeric() with errors='coerce' handles this gracefully: values that cannot be parsed become NaN, which can then be handled by the imputation strategies already in place.

df['revenue'] = pd.to_numeric(df['revenue'], errors='coerce')
# rows that had "N/A" or "--" now have NaN; handle with fillna or drop

Detecting and Removing Duplicates and Outliers

Duplicate rows are a common artefact of data ingestion pipelines that process overlapping time windows, merge files from multiple sources, or retry failed operations without deduplication logic. df.duplicated() returns a boolean Series where True marks every row that is a duplicate of an earlier row. df.drop_duplicates() removes them, keeping the first occurrence by default.

print(f"Duplicate rows: {df.duplicated().sum()}")
df = df.drop_duplicates()

# When only specific columns should be unique (e.g., one row per order_id):
df = df.drop_duplicates(subset=['order_id'], keep='first')

The subset parameter is important: in many business datasets, a full-row duplicate is not expected, but a duplicate on a business key (order ID, customer ID, transaction reference) represents a genuine data error that must be resolved.

Outlier detection with the IQR method is the most widely used non-parametric approach because it makes no assumption about the underlying distribution. The interquartile range is defined as Q3 − Q1 (the 75th percentile minus the 25th percentile). Any value more than 1.5 × IQR below Q1 or above Q3 is flagged as a potential outlier. This threshold is the conventional default; for very strict cleaning, 1.0 × IQR is used; for lenient cleaning, 3.0 × IQR.

Q1  = df['revenue'].quantile(0.25)
Q3  = df['revenue'].quantile(0.75)
IQR = Q3 - Q1

lower_bound = Q1 - 1.5 * IQR
upper_bound = Q3 + 1.5 * IQR

outliers = df[(df['revenue'] < lower_bound) | (df['revenue'] > upper_bound)]
print(f"Outlier rows: {len(outliers)}")

Rather than dropping outlier rows—which reduces the sample size and can introduce selection bias—a common and often preferable strategy is capping (Winsorisation): replacing values below the lower bound with the lower bound and values above the upper bound with the upper bound. Pandas provides clip() for exactly this operation.

# Cap revenue at IQR boundaries, preserving all rows
df['revenue'] = df['revenue'].clip(lower=lower_bound, upper=upper_bound)

NumPy's np.where() offers a more expressive alternative when different replacement logic is needed for different conditions:

df['revenue'] = np.where(df['revenue'] > upper_bound, upper_bound,
                np.where(df['revenue'] < lower_bound, lower_bound,
                         df['revenue']))

The decision to drop versus cap versus transform (e.g., log-transform) outliers should always be informed by visualisation. A box plot immediately shows the spread and position of outliers; a z-score histogram shows how many standard deviations each point lies from the mean. These visual checks are the evidence base for a defensible treatment decision.

import matplotlib.pyplot as plt

# Box plot
df.boxplot(column='revenue')
plt.title('Revenue Distribution')
plt.show()

# Z-score distribution
from scipy import stats
z_scores = np.abs(stats.zscore(df['revenue'].dropna()))
plt.hist(z_scores, bins=50)
plt.axvline(x=3, color='red', linestyle='--', label='z=3 threshold')
plt.legend()
plt.show()

String Cleaning and Feature Extraction

Text columns in real datasets are an almost unlimited source of inconsistency. The same city may be recorded as "New York", "new york", " New York ", and "NEW YORK" in different rows, making every group count on that column wrong. The Pandas .str accessor exposes a rich set of vectorised string methods that apply to every element of a Series without a Python loop.

The most fundamental cleaning operations are whitespace removal and case normalisation, and they should almost always be applied together as the very first transformation on any text column:

df['city'] = df['city'].str.strip().str.lower()

str.replace() accepts both literal substrings and full regular expressions, making it powerful enough to remove currency symbols, percentage signs, parentheses, and other formatting artefacts that prevent type conversion:

# Remove currency symbol and commas before numeric conversion
df['price'] = df['price'].str.replace(r'[$,]', '', regex=True)
df['price'] = pd.to_numeric(df['price'], errors='coerce')

# Remove all non-alphanumeric characters from a product code
df['product_code'] = df['product_code'].str.replace(r'[^a-zA-Z0-9]', '', regex=True)

Extracting structured information from free-text fields is one of the most valuable string operations in practical data work. str.extract() applies a regular expression with named capture groups and returns each group as a new column. Consider a column that contains entries like "Order #12345 - 2024-03-15":

extracted = df['order_notes'].str.extract(
    r'Order #(?P\d+) - (?P\d{4}-\d{2}-\d{2})'
)
df['order_id']   = pd.to_numeric(extracted['order_id'])
df['order_date'] = pd.to_datetime(extracted['order_date'])

Each named group becomes a column in the returned DataFrame. This is far more maintainable than ad-hoc string slicing because the intent is explicit in the pattern itself.

Splitting compound fields is another common task. A column containing full names, addresses, or combined codes can be split into separate, individually queryable columns using str.split() with expand=True, which returns a DataFrame rather than a Series of lists:

name_parts = df['full_name'].str.split(' ', n=1, expand=True)
df['first_name'] = name_parts[0]
df['last_name']  = name_parts[1]

# Split "city, state" into two columns
location_parts = df['location'].str.split(',', expand=True)
df['city']  = location_parts[0].str.strip()
df['state'] = location_parts[1].str.strip()

Reshaping and Aggregating for Analysis-Ready Structure

Cleaning fixes data quality problems, but reshaping addresses structural problems—datasets that are technically valid but organised in a way that makes analysis or visualisation awkward. The two most common structural issues are wide format (too many columns, each representing a measurement at a different time or category) and the need to compute summaries across groups.

Wide to long with melt(): Many datasets arrive in wide format because that is how spreadsheet users naturally organise data—one column per month, one column per product category, and so on. This structure makes it impossible to use groupby effectively because the dimension of interest is encoded in column names rather than values. df.melt() unpivots the DataFrame, collecting the column names into a new variable column and the cell values into a new value column.

# Wide format: one column per quarter
#   customer_id | Q1_revenue | Q2_revenue | Q3_revenue | Q4_revenue

df_long = df.melt(
    id_vars=['customer_id'],
    value_vars=['Q1_revenue', 'Q2_revenue', 'Q3_revenue', 'Q4_revenue'],
    var_name='quarter',
    value_name='revenue'
)
# Result: customer_id | quarter | revenue
# Now groupby('quarter') works naturally

Summary tables with pivot_table(): The inverse operation—collapsing a long-format DataFrame into a compact summary grid—is handled by pivot_table(). It is analogous to a spreadsheet pivot table and supports multiple aggregation functions, multiple index and column dimensions, and automatic handling of missing combinations via the fill_value parameter.

summary = df.pivot_table(
    values='revenue',
    index='region',
    columns='product_category',
    aggfunc='sum',
    fill_value=0
)
print(summary)

Multiple aggregations with groupby().agg(): When the goal is not a pivot grid but rather a flat summary table with multiple statistics per group, passing a dictionary to .agg() computes them all in a single, efficient pass over the data—far more efficient than calling .mean(), .sum(), and .count() separately.

summary = df.groupby('region').agg(
    total_revenue=('revenue', 'sum'),
    avg_revenue=('revenue', 'mean'),
    order_count=('order_id', 'count'),
    unique_customers=('customer_id', 'nunique')
).reset_index()

print(summary)

The named aggregation syntax (a tuple of (source_column, function) assigned to a new column name) produces clean, unambiguous column names in the output. Calling reset_index() at the end promotes the group keys from the index back into regular columns, producing a flat DataFrame that exports cleanly and is immediately usable downstream.

Validating and Exporting the Cleaned Dataset

The final stage of the pipeline is not export—it is validation. Before writing any output file, the cleaned DataFrame should be subjected to a set of programmatic assertions that confirm every invariant the cleaning process was supposed to establish. These checks serve as the automated acceptance criteria for the pipeline.

def validate_clean_dataset(df):
    # No nulls in business-critical columns
    critical_cols = ['order_id', 'customer_id', 'order_date', 'revenue']
    for col in critical_cols:
        null_count = df[col].isnull().sum()
        assert null_count == 0, f"{col} still has {null_count} nulls"

    # Revenue must be non-negative
    assert (df['revenue'] >= 0).all(), "Negative revenue values found"

    # Dates must be within a plausible range
    assert df['order_date'].min() >= pd.Timestamp('2018-01-01'), "Dates too early"
    assert df['order_date'].max() <= pd.Timestamp('2025-12-31'), "Dates in future"

    # Row count should be within expected range after deduplication
    assert 10_000 <= len(df) <= 500_000, f"Unexpected row count: {len(df)}"

    print("All validation checks passed.")

validate_clean_dataset(df)

After assertions pass, comparing the describe() output of the cleaned DataFrame against that of the raw DataFrame gives a distribution-level sanity check. The mean and standard deviation of numeric columns should change only to the extent that imputation and capping are expected to change them; a dramatic shift signals an unintended transformation.

For export, the choice of format depends on the downstream use. CSV is universally readable but has no type information—a datetime column written to CSV comes back as a string on the next read, requiring the type conversion step again. Parquet preserves dtypes exactly (including datetime64, category, and float32), compresses efficiently using columnar encoding, and reads back far faster than CSV for large datasets. Use CSV when the consumer is a human or a non-Python tool; use Parquet for any programmatic downstream pipeline.

# CSV: universally compatible, no type preservation
df.to_csv('sales_clean.csv', index=False)

# Parquet: compressed, type-preserving, fast for large datasets
df.to_parquet('sales_clean.parquet', index=False, compression='snappy')

Finally, the cleaning script or notebook should be committed to version control as a first-class artefact of the project, not treated as a throwaway exploration. A well-documented, version-controlled pipeline means that when the data source changes schema next quarter, the pipeline can be updated in a single place, the change is tracked in git history with a clear commit message, and the same validated logic flows through to every downstream consumer automatically.
