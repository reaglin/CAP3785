Data Transformation with Power Query

▶

Power Query is one of the most powerful data preparation tools built directly into Microsoft Excel and Power BI. Before data can be meaningfully analyzed with pivot tables, charts, or formulas, it almost always needs to be cleaned, reshaped, and standardized. Raw data from real-world sources — exported spreadsheets, database dumps, CSV files from third-party systems — is rarely ready to use as-is. Power Query solves this problem by providing a dedicated, visual environment where you can apply a sequence of transformation steps to your data in a completely controlled, repeatable, and non-destructive way. Once you understand how to work inside this environment, you eliminate hours of tedious manual data preparation and replace it with a process that runs automatically every time your data updates.

Introduction to the Power Query Editor

The Power Query Editor is a separate window that opens when you create or edit a query in Excel (via Data > Get Data or Data > Queries & Connections). Think of it as a dedicated workbench for shaping your data before it ever arrives in your spreadsheet or data model. Unlike working directly on a worksheet, every change you make inside the editor is recorded as a discrete, named step.

These steps accumulate in the Applied Steps pane on the right side of the editor window. Each step represents a single transformation — for example, "Removed Blank Rows," "Changed Type," or "Split Column." You can click on any step to see exactly what the data looked like at that point in the process, making it straightforward to identify where something went wrong or to understand the logic of a query someone else built. This sequential, visible approach is what makes Power Query so much easier to audit and troubleshoot compared to a tangled chain of worksheet formulas.

A critical principle to understand from the start is that all transformations in Power Query are non-destructive. The original data source — whether it is a CSV file on your desktop, a table in a SQL Server database, or a SharePoint list — is never modified. Power Query reads the source, applies your steps in memory, and delivers the result. If you delete a step, the underlying source is completely unaffected. This safety net encourages experimentation: you can try a transformation, see the result, and roll it back instantly by clicking the small X next to the step in the Applied Steps pane.

For users who want to go beyond point-and-click, the formula bar at the top of the editor displays the M language expression that corresponds to the currently selected step. M (also called Power Query Formula Language) is a functional language purpose-built for data transformation. For example, when you change a column's data type to Date, the formula bar will show something like:

= Table.TransformColumnTypes(#"Previous Step Name", {{"OrderDate", type date}})

You can edit this expression directly, which gives you fine-grained control that exceeds what the graphical interface exposes. Most users begin entirely with the visual tools and gradually learn M as their needs grow more complex. Even at the beginner level, understanding that each step is just an M expression demystifies what Power Query is doing under the hood.

Connecting to and Importing Raw Data Sources

The first step in any Power Query workflow is connecting to a data source. Excel's Power Query supports an extensive range of sources, which can be grouped into a few broad categories:

  • Local files: Excel workbooks (.xlsx, .xls), comma-separated values (.csv), text files (.txt), XML files, and JSON files. These are among the most common starting points for business analysts.
  • Relational databases: SQL Server, Microsoft Access, MySQL, PostgreSQL, Oracle, and many others via ODBC drivers. Power Query connects directly and can even pass through custom SQL queries to filter data at the source before importing.
  • Online and cloud services: SharePoint lists, SharePoint folders, Microsoft Dataverse, Power BI datasets, Azure SQL, and various REST API endpoints via the Web connector.
  • Folder sources: Entire folders of files of the same type (for example, twelve monthly CSV exports) can be combined into a single query automatically, with Power Query stacking the files on top of each other.

One of the most practical capabilities is combining multiple sources into a single query. The Append Queries operation stacks tables with the same columns on top of each other — ideal for consolidating monthly reports into an annual dataset. The Merge Queries operation performs a join (similar to a VLOOKUP or SQL JOIN) between two tables on one or more matching key columns, so you can enrich a sales table with product descriptions from a separate lookup table, all within Power Query.

For teams working with regularly changing file locations or wanting to make a query reusable, parameterized connections are a powerful technique. A parameter in Power Query is a named, changeable value — for instance, a file path or a folder location. Instead of hard-coding C:\Reports\January.csv into your query, you reference a parameter called FilePath. When you want to point the query at a different file, you update the parameter value in one place and every query that references it adjusts automatically. This is particularly valuable for template workbooks distributed across a team.

Cleaning Data: Removing Errors, Duplicates, and Blanks

Raw data almost always contains noise that distorts analysis: duplicate records that inflate counts, blank rows that break calculations, and error values that propagate through formulas. Power Query provides dedicated, easy-to-use tools for each of these problems.

Removing duplicates is accomplished by selecting one or more columns (hold Ctrl to select multiple), right-clicking, and choosing Remove Duplicates. Power Query keeps the first occurrence of each unique combination of values in the selected columns and removes all subsequent duplicates. For example, if your customer table was exported twice and contains every customer record twice over, selecting the CustomerID column and removing duplicates instantly resolves the problem. This is far faster and less error-prone than using Excel's built-in Remove Duplicates feature on a worksheet, because it becomes a permanent, automatically re-applied step in your query.

Blank rows and null values are a separate but related problem. A blank row — a row where every cell is empty — typically enters data through poor export formatting and has no analytical value. You can remove all blank rows at once using Home > Remove Rows > Remove Blank Rows. Individual null values within columns (shown as null in the editor) can be handled in two ways: you can filter them out entirely using the column's dropdown filter, or you can replace them with a meaningful default by right-clicking the column and selecting Replace Values, entering null as the value to find and your default (such as 0 or Unknown) as the replacement. Replacing nulls with zero in numeric columns is especially important before loading data into a pivot table, where null values can cause totals to appear lower than expected or cause certain aggregation functions to behave unexpectedly.

Error values — cells that contain an error rather than a valid data value — often arrive when you import data from a source system that has its own calculation errors (such as division by zero represented as #DIV/0! in an exported Excel file, which Power Query reads as an error type). Left uncleaned, these errors propagate and cause problems downstream. You can handle them using Transform > Replace Errors, which lets you specify a replacement value (commonly null, 0, or a text label like "Error") for any error in the selected column. Alternatively, you can remove entire rows that contain errors using Home > Remove Rows > Remove Errors, which is appropriate when a row with an error is genuinely unusable rather than salvageable with a default value.

Reshaping Data: Splitting, Merging, and Pivoting Columns

Beyond cleaning, data often needs to be restructured — its shape changed — before it is useful. Power Query's reshaping tools let you split compound values into separate columns, combine multiple columns into one, and convert between wide and tall table formats.

Split Column is used when a single column contains multiple pieces of information jammed together. A classic example is a "Full Name" column containing values like Smith, John. Using Transform > Split Column > By Delimiter, you specify the comma as the delimiter and Power Query creates two new columns: Full Name.1 (containing Smith) and Full Name.2 (containing John), which you then rename to LastName and FirstName. You can also split by a fixed number of characters — useful for codes like US-2024-00142 where each segment has a known length — or by the transition between lowercase and uppercase letters for camel-cased identifiers.

Merge Columns does the opposite: it takes two or more existing columns and concatenates their values into a single column with a separator you choose. For instance, merging a City column and a Country column with a comma-space separator would produce values like Berlin, Germany. This is useful for creating display labels or composite keys.

Unpivot Columns is arguably the most transformative reshaping operation, and one of the most important to understand. It converts a wide-format table — where multiple column headers represent categories or time periods — into a tall-format (also called normalized or long-format) table, which has one row per observation. Consider this example of a wide-format sales table:

Product January February March
Widget A 1200 1500 1100
Widget B 800 950 870

In this layout, the month names are column headers, which makes it visually readable but analytically rigid — you cannot easily filter by month, and a pivot table cannot group or sum across months as a dimension. After selecting the January, February, and March columns and applying Unpivot Columns, the result becomes:

Product Attribute Value
Widget A January 1200
Widget A February 1500
Widget A March 1100
Widget B January 800
Widget B February 950
Widget B March 870

Now the month is a proper data dimension in the Attribute column (which you would rename to Month), and the sales figure is consistently in the Value column (renamed to Sales). This tall-format table is exactly what a pivot table needs to slice and dice sales by product and by month simultaneously.

Pivot Column is the reverse operation: it takes a column of unique values and turns each distinct value into its own column header, populating each new column with an aggregated value from another column. This is useful for creating summary views inside Power Query itself, though most analysts use pivot tables in Excel for this purpose and reserve Power Query's Pivot Column for specific structural transformations mid-query.

Changing Data Types and Formatting Values

Every column in Power Query has a data type, indicated by an icon in the column header — an ABC icon for text, 123 for whole numbers, a calendar icon for dates, and so on. Setting the correct data type is not optional: it is foundational to everything else working correctly.

To change a column's type, click the type icon at the top left of the column header and select from the list: Whole Number, Decimal Number, Text, Date, Date/Time, Date/Time/Timezone, Duration, True/False (Boolean), or Binary. When you import a CSV file, Power Query will attempt to auto-detect types in an automatically inserted "Changed Type" step, but this auto-detection is not always correct and should always be verified. A column of product codes like 00145 might be incorrectly inferred as a Whole Number (losing the leading zeros) when it should remain Text.

Date columns deserve special attention. A column of dates stored as text (for example, the string "2024-03-15") looks like a date but is not one. Calculations, sorting, and time-intelligence functions in pivot tables will all fail or produce incorrect results unless the column is typed as Date or Date/Time. Once correctly typed, Excel pivot tables can group dates by Year, Quarter, Month, and Day automatically, and Power BI's time-intelligence DAX functions will work as expected.

Beyond types, the Format options under the Transform tab let you standardize the content of text columns:

  • Trim: Removes leading and trailing spaces. This is essential when text has been entered or exported with inconsistent spacing, because "London" and " London" will be treated as two different values in a pivot table.
  • Clean: Removes non-printable characters — invisible characters that sometimes appear in data exported from legacy systems and that can cause lookups and comparisons to silently fail.
  • Uppercase / Lowercase / Capitalize Each Word: Standardizes casing so that "SMITH", "smith", and "Smith" all become the same value. This is particularly important for joining tables on name fields, where inconsistent casing would cause matches to fail.

Adding Custom and Conditional Columns

Sometimes the columns that exist in your source data are not sufficient — you need to derive new values from them. Power Query provides two primary ways to add calculated columns without touching the source data.

Custom Column (accessed via Add Column > Custom Column) opens a dialog where you write an M language expression. The result of the expression becomes the value for that column in every row. For example, to calculate a total price from a Quantity column and a UnitPrice column, your expression would be:

[Quantity] * [UnitPrice]

Column names are referenced in square brackets. You can also write more complex expressions — for instance, to concatenate a first name and last name with a space:

[FirstName] & " " & [LastName]

Or to calculate a discounted price conditionally, using M's if expression directly in the custom column formula:

if [Quantity] >= 100 then [UnitPrice] * 0.9 else [UnitPrice]

Conditional Column (accessed via Add Column > Conditional Column) provides a point-and-click interface for building if-then-else logic without writing M code. You define one or more conditions — for example, if [Region] equals "North" then "Northern Division" — and specify what value to return when each condition is met and what default to return otherwise. This is functionally equivalent to Excel's nested IF formula, but applied across the entire dataset automatically through Power Query rather than needing to be dragged down through rows on a worksheet.

The key advantage of both approaches is automatic refresh coverage. Because the column logic lives inside the query, it is recalculated across the entire dataset every time the query refreshes — including any new rows that have been added to the source since the last refresh. Compare this to a worksheet formula in column H that you have to remember to copy down to row 5000 every time you get new data: Power Query eliminates that manual step entirely and removes the risk of a formula being missing from some rows.

Refreshing Queries and Automating Data Preparation

A query that you have meticulously built and configured is only as valuable as its ability to stay current. Power Query's refresh mechanism is what transforms a one-time data cleaning exercise into a repeatable, automated data pipeline.

To manually refresh a query or all queries in a workbook, navigate to the Data tab in Excel and click Refresh (for the selected query) or Refresh All (to refresh every query and connection in the workbook simultaneously). Power Query re-reads the data source(s), re-applies every step in the Applied Steps sequence, and delivers the updated, transformed result to its output destination — either a worksheet table or the Excel Data Model.

For workbooks that need to stay up to date without manual intervention, you can configure automatic refresh on open. Right-click a query in the Queries & Connections pane and select Properties. Under the Usage tab, check Refresh data when opening the file. From that point on, every time the workbook is opened, Power Query will automatically re-read and re-transform the source data. This is ideal for daily dashboards where the underlying CSV or database records are updated overnight — the analyst simply opens the workbook each morning and sees fresh data.

A common concern with automated pipelines is schema changes in the source — situations where the source data structure changes, such as a new column being added to a database export, a column being renamed, or a file being saved in a slightly different format. When this happens, Power Query will often display errors in the Applied Steps pane pointing to the step where the schema assumption broke down. Because the query's logic is transparent and editable step by step, you can navigate to the problematic step, update it to reflect the new structure, and restore the query to working order without rebuilding it from scratch. For example, if a column called CustomerName is renamed Client_Name in the source, you can edit the specific step in the formula bar to reference the new name rather than recreating all the downstream transformations that depend on it.

Understanding this refresh and maintenance model positions Power Query not just as a cleanup tool, but as the foundation of a robust, low-maintenance data preparation layer — one that handles the messy, repetitive work of turning raw source data into analysis-ready datasets automatically, every time.
