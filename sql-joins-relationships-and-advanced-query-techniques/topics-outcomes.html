Module Topics

Relational Database Foundations

Introduces the concept of related tables and how foreign keys establish connections between them. Sets the foundation for understanding why JOIN operations are necessary in multi-table databases.
  • What Is a Relational Database? — A relational database organizes data into multiple structured tables, each representing a distinct entity or concept.
  • Primary Keys: Uniquely Identifying Records — A primary key is a column or set of columns that uniquely identifies each row within a table.
  • Foreign Keys: Creating Relationships Between Tables — A foreign key is a column in one table that references the primary key of another table, establishing a link between the two.
  • Types of Table Relationships — Relationships between tables are categorized by how many records on each side can be associated with records on the other side.
  • Why Data Is Split Across Multiple Tables — Storing all data in a single table leads to repetition, inconsistency, and difficulty maintaining accurate records over time.
  • The Need for JOIN Operations — Because related data lives in separate tables, queries that require information from more than one table must use JOIN operations to combine them.

INNER JOIN and Matching Records

Covers the INNER JOIN operation to retrieve only records with matching values in both tables. Explores common use cases and syntax for combining related data sets.
  • What Is an INNER JOIN? — An INNER JOIN combines rows from two or more tables based on a related column, returning only the rows where a match exists in both tables.
  • Basic INNER JOIN Syntax — The INNER JOIN clause is placed in the FROM section of a SELECT statement, followed by an ON condition that specifies which columns to match.
  • Matching on Primary and Foreign Keys — INNER JOIN most often links a foreign key in one table to the primary key in another, reflecting the underlying relational data model.
  • Using Table Aliases for Readability — Aliases assign short, readable names to tables within a query, simplifying the syntax when referencing columns from multiple tables.
  • Filtering INNER JOIN Results with WHERE — A WHERE clause can be appended after an INNER JOIN to further narrow the result set based on additional conditions beyond the join predicate.
  • Common Use Cases for INNER JOIN — INNER JOIN is used whenever analysis requires only records that have confirmed relationships across two or more tables.
  • Joining More Than Two Tables — Multiple INNER JOIN clauses can be chained in a single query to combine three or more related tables simultaneously.

Outer JOINs: LEFT, RIGHT, and FULL

Examines LEFT, RIGHT, and FULL OUTER JOIN operations and how they handle unmatched records from one or both tables. Highlights practical differences and when to apply each type.
  • What Is an Outer JOIN? — An OUTER JOIN retrieves rows from one or both tables even when no matching record exists in the other table, unlike INNER JOIN which only returns matched rows.
  • LEFT OUTER JOIN — A LEFT OUTER JOIN returns all rows from the left (first) table and only the matching rows from the right table, filling unmatched right-side columns with NULL.
  • RIGHT OUTER JOIN — A RIGHT OUTER JOIN mirrors the LEFT JOIN but preserves all rows from the right (second) table, filling unmatched left-side columns with NULL.
  • FULL OUTER JOIN — A FULL OUTER JOIN combines the behavior of LEFT and RIGHT JOINs, returning all rows from both tables and filling NULLs wherever no match exists on either side.
  • Handling NULL Values in Outer JOIN Results — Because OUTER JOINs introduce NULL values for unmatched rows, queries often need to account for NULLs in filtering and calculation logic.
  • Practical Differences and Choosing the Right Outer JOIN — Selecting the correct OUTER JOIN type depends on which dataset must remain complete and what business question is being answered.

Subqueries and Nested SELECT Statements

Introduces subqueries as a technique for embedding one SELECT statement inside another to filter, compute, or compare data. Covers both correlated and non-correlated subquery patterns.
  • What Is a Subquery? — A subquery is a SELECT statement nested inside another SQL statement, allowing you to use the result of one query as input to another.
  • Subqueries in the WHERE Clause — The most common use of a subquery is inside a WHERE clause to filter rows based on values computed from another table or query.
  • Subqueries in the FROM Clause (Derived Tables) — A subquery placed in the FROM clause creates a temporary result set, often called a derived table or inline view, that the outer query can reference.
  • Non-Correlated Subqueries — A non-correlated subquery is independent of the outer query and executes only once, with its result reused for every row evaluated by the outer query.
  • Correlated Subqueries — A correlated subquery references columns from the outer query, causing it to execute once for every row processed by the outer query.
  • Using EXISTS and NOT EXISTS — The EXISTS operator tests whether a correlated subquery returns at least one row, and NOT EXISTS tests whether it returns no rows.
  • Subqueries vs. JOINs: Choosing the Right Approach — Both subqueries and JOINs can solve many of the same data retrieval problems, but they differ in readability, performance, and capability.

Aggregation with GROUP BY and HAVING

Explores how to summarize data using aggregate functions alongside the GROUP BY clause to group results by one or more columns. Introduces the HAVING clause for filtering aggregated results after grouping.
  • Introduction to Aggregate Functions — Aggregate functions perform calculations on a set of rows and return a single summary value, making them essential for data analysis in SQL.
  • Grouping Data with GROUP BY — The GROUP BY clause divides rows into groups based on one or more columns, allowing aggregate functions to be applied to each group independently.
  • Combining Aggregate Functions with GROUP BY — Pairing aggregate functions with GROUP BY produces meaningful per-group summaries, enabling comparisons across categories, departments, time periods, or any other dimension.
  • Filtering Grouped Results with HAVING — The HAVING clause filters the results of a GROUP BY query based on conditions applied to aggregated values, similar to how WHERE filters individual rows.
  • WHERE vs. HAVING: Knowing When to Use Each — Understanding the difference between WHERE and HAVING is critical for writing correct aggregation queries, as each operates at a different stage of query execution.
  • Ordering and Presenting Aggregated Results — The ORDER BY clause can be combined with GROUP BY and HAVING to sort the final aggregated output in a meaningful way for analysis or reporting.

Solving Multi-Table Analytical Problems

Applies JOIN operations, subqueries, and aggregation together to solve complex, real-world analytical queries across multiple tables. Reinforces decision-making around query design and technique selection.
  • Decomposing a Complex Analytical Question — Before writing a single line of SQL, it is essential to break a complex business question into smaller, answerable sub-problems that map to specific tables and operations.
  • Choosing the Right JOIN Type for the Problem — Selecting the appropriate JOIN type is a critical design decision that directly affects which rows appear in the result set and whether the analysis is accurate.
  • Layering Aggregation with GROUP BY and HAVING — Multi-table analytical queries frequently require summarizing data after joining, using GROUP BY to segment results and HAVING to filter aggregated groups.
  • Using Subqueries to Stage Intermediate Results — Subqueries allow you to pre-process or filter data in one step and feed that result into a larger query, making complex logic easier to manage and reason about.
  • Combining JOINs and Subqueries in a Single Query — Real-world analytical queries often require both JOINs and subqueries working together, where a derived table or filtered subquery serves as one of the inputs to a JOIN.
  • Validating and Interpreting Multi-Table Query Results — After a complex query runs, verifying that the results are logically correct is just as important as writing syntactically valid SQL.
  • Query Design Decision-Making Framework — Selecting the right combination of techniques — joins, subqueries, aggregation — requires a systematic approach to weigh readability, correctness, and performance against the requirements of the problem.
