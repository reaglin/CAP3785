Subqueries and Nested SELECT Statements

▶

One of the most powerful features of SQL is the ability to embed one SELECT statement inside another. This technique is called a subquery, and it opens up a wide range of filtering, computation, and comparison possibilities that would be difficult or impossible to express in a single, flat query. Understanding subqueries — how they execute, where they can appear, and when to use them — is a cornerstone skill for anyone working with relational databases.

A subquery (also called an inner query or nested query) is a complete SELECT statement enclosed in parentheses that is embedded within another SQL statement. The outer statement that contains it is called the outer query. The fundamental rule governing execution order is: the inner query always executes first, and its result is then handed to the outer query to use. Think of a subquery as a question whose answer is immediately fed into a larger question.

Subqueries can return different shapes of data depending on context:

  • A single scalar value — one row, one column (e.g., the maximum salary in a table)
  • A single row — one row with multiple columns
  • A result set (table) — multiple rows and columns, often used as a derived table
  • A list of values — a single column with multiple rows, used with IN

Because the inner query runs independently before the outer query evaluates its conditions, subqueries let you perform complex, multi-step logic inside a single SQL statement without needing to store intermediate results in temporary tables or run multiple separate queries.

Subqueries in the WHERE Clause are the most common form. Here the subquery produces a value or set of values that the outer query uses as a filter condition. The right operator to use depends on what the subquery returns.

When the subquery returns exactly one value (a scalar), you can use standard comparison operators such as =, >, <, >=, <=, or <>. For example, suppose you have an employees table and you want to find every employee who earns exactly the average salary:

SELECT first_name, last_name, salary
FROM employees
WHERE salary = (SELECT AVG(salary) FROM employees);

The inner query SELECT AVG(salary) FROM employees computes a single number — say, 55000. The outer query then filters rows where salary = 55000. If the subquery could return more than one row and you use =, the database engine will raise an error, so it is important to know your data and ensure scalar subqueries truly return only one value.

When the subquery returns a list of values (a single column, many rows), use the IN operator. Suppose you want to find all orders placed by customers who live in a specific region, and that region information is in a separate customers table:

SELECT order_id, order_date, amount
FROM orders
WHERE customer_id IN (
    SELECT customer_id
    FROM customers
    WHERE region = 'Northwest'
);

The subquery produces a list of customer_id values; the outer query then returns only those orders whose customer_id appears in that list. NOT IN works similarly, returning rows where the value is absent from the list — though be careful with NOT IN when the subquery result might contain NULL values, because any comparison with NULL using NOT IN causes the entire condition to evaluate as unknown, potentially returning zero rows.

The EXISTS operator takes a different approach: it does not compare a column value at all. Instead, it simply checks whether the subquery returns any rows whatsoever. The outer query includes the current row if the subquery produces at least one row. This is covered in depth below.

Subqueries in the FROM Clause produce what is called a derived table (or inline view). The entire subquery is treated as if it were a real table, and the outer query selects from it. A critical requirement is that the derived table must be given an alias; without an alias the database engine cannot reference its columns.

This pattern is especially useful when you need to filter on the result of an aggregate function. You cannot use WHERE salary > AVG(salary) directly because WHERE is evaluated before aggregation. However, you can compute the aggregation first in a subquery and then filter in the outer query:

SELECT department_id, avg_salary
FROM (
    SELECT department_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
) AS dept_averages
WHERE avg_salary > 60000;

Here the subquery dept_averages computes the average salary per department and gives it a name. The outer query then applies a WHERE filter on that computed column — something impossible to do in the same query without this layered approach. Derived tables make complex logic readable by breaking it into distinct, named steps.

Non-Correlated Subqueries are the simpler of the two main categories. In a non-correlated subquery, the inner query is completely self-contained — it contains no reference to any column from the outer query. This means the database executes the inner query once, produces a fixed result, and then uses that result throughout the evaluation of the outer query. The examples shown so far (finding employees who earn the average salary, finding orders for Northwest customers) are all non-correlated.

A classic use case is finding rows that match a global extreme value. To retrieve the product with the highest unit price:

SELECT product_name, unit_price
FROM products
WHERE unit_price = (SELECT MAX(unit_price) FROM products);

The subquery runs once, returns a single maximum price, and then the outer query filters all products against that single number. Non-correlated subqueries are generally efficient because the inner query is evaluated only once regardless of how many rows the outer query processes.

Correlated Subqueries are fundamentally different in execution. In a correlated subquery, the inner query contains a reference to a column from the outer query. This means the subquery cannot run independently — it depends on the current row being evaluated by the outer query. The database conceptually executes the subquery once for each row in the outer query, passing in values from that row each time.

Consider finding all employees who earn more than the average salary of their own department (not the company-wide average). Because the comparison must be department-specific and each row belongs to a different department, this is a natural fit for a correlated subquery:

SELECT e1.first_name, e1.last_name, e1.department_id, e1.salary
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);

For each row in the outer query (aliased e1), the subquery uses e1.department_id to compute the average salary for that specific department. The subquery result changes row by row. This row-level, context-sensitive logic is the defining strength of correlated subqueries.

The trade-off is performance. Because the subquery may be re-executed for every row in the outer table, correlated subqueries can be slow on large datasets. However, modern database engines often optimize them intelligently (for example, by caching subquery results for repeated values of the correlated column), so actual performance depends on the engine and the data distribution. Always examine the query execution plan when performance matters.

Using EXISTS and NOT EXISTS builds directly on the correlated subquery concept. EXISTS is a logical test: it evaluates to TRUE if the subquery returns one or more rows, and FALSE if it returns no rows. Crucially, EXISTS does not care about the actual column values returned — it only cares about row existence. For this reason it is common (and idiomatic) to write SELECT 1 or SELECT * inside an EXISTS subquery, since the values themselves are discarded.

-- Find customers who have placed at least one order
SELECT customer_id, company_name
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.customer_id
);

For each customer row, the subquery checks whether any row in orders has a matching customer_id. If yes, the customer is included in the result. The database engine can short-circuit as soon as it finds the first matching row, which can make EXISTS faster than IN when the subquery would return a large list.

NOT EXISTS inverts the logic and is one of the cleanest ways to perform an anti-join — finding records in one table that have no corresponding records in another:

-- Find customers who have never placed an order
SELECT customer_id, company_name
FROM customers c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.customer_id
);

This is equivalent in intent to WHERE customer_id NOT IN (SELECT customer_id FROM orders), but with an important advantage: NOT EXISTS is null-safe. If the orders table contains a row where customer_id is NULL, NOT IN will return no results at all because comparing any value to NULL with NOT IN yields unknown, which fails the filter. NOT EXISTS, by contrast, is not affected by NULL values in the subquery and behaves correctly.

Short-circuits on first match? No — full list evaluated Yes — stops at first match Depends on engine
NULL-safe for anti-pattern? No — NOT IN breaks on NULLs Yes — NOT EXISTS is safe Yes (with LEFT JOIN + IS NULL)

Choosing Between Subqueries and JOINs is a practical decision that depends on readability, the shape of the output, and performance characteristics. The key considerations are:

  • Use a JOIN when you need columns from multiple tables in the same result set. JOINs are designed for combining data, and the optimizer has decades of tuning around join algorithms (nested loop, hash join, merge join). If you need customer.name and order.amount side by side, a JOIN is the natural and usually most efficient tool.
  • Use a subquery when the inner logic is purely for filtering or scalar computation. If you only need data from one table but need to filter it based on a condition derived from another table, a subquery can be cleaner and more expressive than a JOIN with a WHERE clause or a HAVING clause.
  • Use a correlated subquery for row-level comparisons that are difficult to express with a JOIN. The "each employee compared to their own department's average" example above is awkward with a JOIN and natural with a correlated subquery.
  • Use execution plans to decide when performance is critical. Both approaches can be rewritten by the query optimizer into equivalent execution plans in many cases. Run EXPLAIN (or its equivalent in your database) to see what the engine actually does, and benchmark with realistic data volumes before assuming one approach is always faster.

To illustrate the equivalence and the tradeoffs, here is the same logical question — "find customers who have placed at least one order" — written three ways:

-- Using IN
SELECT customer_id, company_name
FROM customers
WHERE customer_id IN (SELECT customer_id FROM orders);

-- Using EXISTS
SELECT customer_id, company_name
FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id);

-- Using JOIN with DISTINCT
SELECT DISTINCT c.customer_id, c.company_name
FROM customers c
INNER JOIN orders o ON c.customer_id = o.customer_id;

All three return the same logical result. The JOIN version requires DISTINCT because a customer with multiple orders would appear multiple times without it. The EXISTS version is null-safe and potentially short-circuits early. The IN version is the most concise but must materialize the full list from orders. Which one is "best" depends on your database engine, your data, and the context in which the query lives.

Mastering subqueries means understanding not just the syntax but the logic flow: how results propagate from inner to outer, how correlated references tie executions together row by row, and how to choose the right tool for the problem at hand. Subqueries, derived tables, and the EXISTS family together form a flexible toolkit that complements — and sometimes surpasses — what joins alone can express.
