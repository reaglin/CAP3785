Outer JOINs: LEFT, RIGHT, and FULL

▶

When querying relational databases, it is common to need data from multiple tables simultaneously. The INNER JOIN is the most familiar join type, but it only returns rows where a matching record exists in both tables. In many real-world scenarios this is too restrictive — you need to see every row from at least one table even when no corresponding row exists in the other. This is precisely what OUTER JOINs accomplish. The SQL standard defines three outer join variants: LEFT OUTER JOIN, RIGHT OUTER JOIN, and FULL OUTER JOIN. Each variant controls which table's rows are guaranteed to appear in the result set, filling the absent side's columns with NULL wherever no match is found.

What Is an Outer JOIN?

An OUTER JOIN extends the behavior of an INNER JOIN by including rows that have no match in the joined table. Where a matching row is absent, the database engine inserts NULL into every column that would have come from the missing side. This design means you never lose rows from the "preserved" table, making outer joins essential for tasks like auditing completeness, identifying gaps, and producing reports that must account for every entity regardless of associated activity.

The three key properties shared by all outer joins are:

  • They always return at least every row from the designated "preserved" table (or both tables in the case of FULL OUTER JOIN).
  • Columns from the "non-preserved" side carry NULL wherever no matching row was found.
  • Choosing the correct type depends entirely on which table's completeness is the analytical priority.

LEFT OUTER JOIN

A LEFT OUTER JOIN (written as LEFT JOIN or LEFT OUTER JOIN — the keyword OUTER is optional and the two forms are completely equivalent in standard SQL) guarantees that every row in the left table appears in the result set. When a left-table row has at least one matching row in the right table, those matches are included normally. When a left-table row has no match in the right table, it still appears in the result, but all columns originating from the right table are set to NULL.

Using the sample tables, the following query lists every customer along with any orders they have placed:

SELECT
    c.customer_id,
    c.customer_name,
    o.order_id,
    o.order_total
FROM customers AS c
LEFT JOIN orders AS o
    ON c.customer_id = o.customer_id;

A very common use of LEFT JOIN is to find left-table rows that have no related right-table record — for instance, customers who have never placed an order:

SELECT
    c.customer_id,
    c.customer_name
FROM customers AS c
LEFT JOIN orders AS o
    ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;

Result:

By filtering with WHERE o.order_id IS NULL after the join, only the rows where the right-table column remained NULL (i.e., no match existed) are kept. This pattern — sometimes called an anti-join — is one of the most practical and frequently used query patterns in SQL.

RIGHT OUTER JOIN

A RIGHT OUTER JOIN (equivalently RIGHT JOIN) is the mirror image of a LEFT JOIN. It guarantees that every row in the right table appears in the result, regardless of whether a matching row exists in the left table. When no left-table match is found, the left-table columns contain NULL.

SELECT
    c.customer_id,
    c.customer_name,
    o.order_id,
    o.order_total
FROM customers AS c
RIGHT JOIN orders AS o
    ON c.customer_id = o.customer_id;

Result:

1Alice101250.00
2Bob10289.50
2Bob103310.00
NULLNULL10445.00

Now every order is guaranteed to appear. Order 104 references a non-existent customer, so the customer columns show NULL, but the order itself is not dropped. Carol and David, who have no orders, do not appear because the right table (orders) drives the result — rows without a right-table record are excluded.

An important practical note: a RIGHT JOIN is logically equivalent to swapping the table order and using a LEFT JOIN. The following query produces an identical result to the RIGHT JOIN above:

SELECT
    c.customer_id,
    c.customer_name,
    o.order_id,
    o.order_total
FROM orders AS o
LEFT JOIN customers AS c
    ON c.customer_id = o.customer_id;

Because of this equivalence, many development teams adopt a convention of writing everything as a LEFT JOIN (adjusting table order as needed) to keep query style consistent and easier to read. RIGHT JOIN is most naturally chosen when the schema or query context makes the right-table position the more logical "primary" dataset — for example, when the right table is a reference or lookup table that must not lose rows.

FULL OUTER JOIN

A FULL OUTER JOIN combines LEFT and RIGHT JOIN behavior: every row from both tables is included in the result. Where matches exist, columns from both sides are populated normally. Where no match exists on the left side, the left-table columns show NULL; where no match exists on the right side, the right-table columns show NULL.

SELECT
    c.customer_id   AS cust_id,
    c.customer_name,
    o.order_id,
    o.customer_id   AS order_cust_id,
    o.order_total
FROM customers AS c
FULL OUTER JOIN orders AS o
    ON c.customer_id = o.customer_id;

Result:

Every customer appears (including those without orders) and every order appears (including the one referencing a non-existent customer). This makes FULL OUTER JOIN ideal for reconciliation and gap analysis across two datasets — for example, comparing a list of registered students against enrolled courses to find students not in any course and courses with no students, all in one query.

Not all database engines support FULL OUTER JOIN natively. MySQL, for instance, does not. In those systems, the equivalent result can be produced by taking a UNION of a LEFT JOIN and a RIGHT JOIN:

-- Emulating FULL OUTER JOIN in MySQL
SELECT c.customer_id, c.customer_name, o.order_id, o.order_total
FROM customers AS c
LEFT JOIN orders AS o ON c.customer_id = o.customer_id

UNION

SELECT c.customer_id, c.customer_name, o.order_id, o.order_total
FROM customers AS c
RIGHT JOIN orders AS o ON c.customer_id = o.customer_id;

UNION (without ALL) removes duplicate rows, so any rows that matched on both sides appear only once, just as in a true FULL OUTER JOIN. Databases such as PostgreSQL, SQL Server, Oracle, and DB2 support FULL OUTER JOIN directly.

Handling NULL Values in Outer JOIN Results

Understanding how NULL behaves in outer join results is critical to writing correct queries. Several common pitfalls and techniques deserve careful attention:

  • IS NULL vs. = NULL: SQL's three-valued logic means that NULL = NULL evaluates to unknown, not true. A WHERE clause written as WHERE o.order_id = NULL will never match any row. Always use IS NULL or IS NOT NULL when testing for the absence or presence of a value.
  • Filtering in WHERE vs. ON: Placing a filter condition on the right-table column inside the WHERE clause after a LEFT JOIN effectively converts it back into an INNER JOIN, because WHERE is applied after the join and will exclude the NULL rows. If the intent is to filter right-table rows before joining while still preserving all left-table rows, the condition belongs in the ON clause.
-- This converts the LEFT JOIN into an effective INNER JOIN:
SELECT c.customer_name, o.order_total
FROM customers AS c
LEFT JOIN orders AS o ON c.customer_id = o.customer_id
WHERE o.order_total > 100;   -- excludes Carol and David (their order_total is NULL)

-- This preserves all customers, filtering orders within the join:
SELECT c.customer_name, o.order_total
FROM customers AS c
LEFT JOIN orders AS o
    ON c.customer_id = o.customer_id
    AND o.order_total > 100;  -- Carol and David still appear with NULL order_total
  • COUNT() and NULLs: COUNT(*) counts every row in the result including those with NULL in joined columns, while COUNT(column) counts only non-NULL values in that column. This distinction is extremely useful when computing summaries over outer join results.
SELECT
    c.customer_name,
    COUNT(*)             AS total_result_rows,
    COUNT(o.order_id)    AS actual_orders
FROM customers AS c
LEFT JOIN orders AS o ON c.customer_id = o.customer_id
GROUP BY c.customer_name;
customer_nametotal_result_rowsactual_orders
Alice11

David10

Carol and David correctly show 0 actual orders because COUNT(o.order_id) ignores their NULL order_id values, while COUNT(*) still counts their single result row.

  • Replacing NULLs for display: Use COALESCE(expression, fallback) or database-specific functions like IFNULL() (MySQL) or ISNULL() (SQL Server) to substitute a meaningful default value in place of NULL in output columns.
SELECT
    c.customer_name,
    COALESCE(o.order_total, 0.00) AS order_total
FROM customers AS c
LEFT JOIN orders AS o ON c.customer_id = o.customer_id;

This returns 0.00 rather than NULL for Carol and David, which is often more readable in reports and avoids confusion for end users unfamiliar with SQL's treatment of NULL.

Practical Differences and Choosing the Right Outer JOIN

Selecting the appropriate join type is fundamentally a question of which records must not be lost. A concise decision framework:

  • Use LEFT JOIN when the left table is your primary dataset and you must retain every one of its rows — for example, listing every employee with or without a department assignment, every product with or without a recent sale, or every student with or without a grade.
  • Use RIGHT JOIN when the right table is the primary or reference dataset and its completeness must be preserved — for example, ensuring every scheduled shift appears in a report even if no staff member is assigned. In practice, many teams simply reverse the table order and use LEFT JOIN instead, which is functionally identical.
  • Use FULL OUTER JOIN when both datasets are equally authoritative and you need to see all rows from both sides simultaneously — for example, comparing two import files to find records that appear in one but not the other, or reconciling a master list against a transaction log.
  • Use INNER JOIN when only rows with matches on both sides are meaningful — for example, retrieving only orders that have valid, existing customers.

The clearest analytical signal that an OUTER JOIN is needed is the requirement to identify missing or unmatched records. If the business question contains phrases like "with or without," "even if there are none," "including those that have not," or "find customers who never ordered," an OUTER JOIN is almost certainly the right tool. An INNER JOIN silently discards non-matching rows, which can produce misleadingly incomplete results if the absence of a relationship is itself meaningful information.

Mastering outer joins — and especially recognizing when the absence of a match is the meaningful data point — is one of the most impactful skills in SQL. Whether auditing referential integrity, building comprehensive reports, or reconciling disparate data sources, LEFT, RIGHT, and FULL OUTER JOINs provide the precise control needed to ensure no important row is silently excluded from your results.
