Handling Missing Data

▶

Missing data is one of the most common and consequential problems encountered in real-world data analysis. Whether caused by sensor failures, survey non-responses, data entry errors, or system outages, missing values can silently distort statistical summaries, bias machine learning models, and produce misleading visualizations. Pandas provides a rich set of tools for detecting, understanding, removing, and filling missing data, allowing you to make deliberate, defensible decisions about how to handle gaps before your analysis begins. The strategies you choose should be informed by the nature and extent of the missingness, the type of data involved, and the downstream use of your results.

Detecting Missing Values

The first step in handling missing data is knowing where it exists and how much of it you have. Pandas makes this straightforward with a pair of complementary methods. df.isnull() returns a boolean DataFrame of the same shape as the original, where every cell that contains a missing value is marked True and every present value is marked False. Its inverse, df.notnull(), returns True wherever a value is present. These boolean DataFrames can themselves be used for filtering or masking operations.

import pandas as pd
import numpy as np

data = {
    'name':  ['Alice', 'Bob', None, 'Diana'],
    'age':   [25, np.nan, 30, 22],
    'score': [88.0, 91.0, np.nan, np.nan]
}
df = pd.DataFrame(data)

print(df.isnull())
#     name    age  score
# 0  False  False  False
# 1  False   True  False
# 2   True  False   True
# 3  False  False   True

To get a practical count of how many values are missing per column, chain .sum() after .isnull(). Because Python treats True as 1 and False as 0, summing the boolean values produces an integer count of missing entries in each column.

print(df.isnull().sum())
# name     1
# age      1
# score    2
# dtype: int64

For larger datasets with many columns, raw counts can be hard to interpret. Computing df.isnull().mean() instead gives the proportion of missing values per column, expressed as a float between 0 and 1. This is immediately actionable: a column where 5% of values are missing calls for a different strategy than one where 70% are missing.

print(df.isnull().mean())
# name     0.25
# age      0.25
# score    0.50
# dtype: float64

You can also get a full overview of the DataFrame including non-null counts by calling df.info(), which prints a concise summary of each column's dtype and the number of non-null entries alongside the total row count.

Understanding NaN and None in NumPy and Pandas

Before deciding how to handle missing values, it is important to understand the different sentinels Pandas uses to represent them, because they behave differently depending on the column's data type.

NumPy's np.nan is a special IEEE 754 floating-point value that stands for "Not a Number." It is the standard representation for missing data in numerical columns. One critical property of np.nan is that it is not equal to itself — np.nan == np.nan evaluates to False. This means you can never use equality checks to find NaN values; you must use pd.isnull() or np.isnan() instead. Furthermore, any arithmetic operation involving np.nan propagates NaN through the calculation by default:

print(5 + np.nan)   # nan
print(np.nan * 0)   # nan  — even multiplying by zero gives nan
print(np.mean([1, 2, np.nan]))  # nan

This propagation can silently corrupt summary statistics. Pandas aggregation functions like .mean(), .sum(), and .std() skip NaN by default (via skipna=True), which is usually helpful, but raw NumPy operations do not.

Python's None is an object-type missing value. In columns with dtype object (typically string columns), Pandas treats None and np.nan as equivalent missing value representations. In numeric columns, assigning None will cause Pandas to upcast the column to float64 and store it as np.nan automatically, because a pure integer array cannot hold a floating-point sentinel.

s = pd.Series([1, 2, None])
print(s.dtype)   # float64  — Pandas upcasted the column
print(s)
# 0    1.0
# 1    2.0
# 2    NaN

Pandas 1.0 introduced pd.NA as a more consistent and type-aware missing value indicator. Unlike np.nan, pd.NA works correctly with integer, boolean, and string extension array types (e.g., Int64, boolean, StringDtype) without forcing a dtype upcast. This means you can now have an integer column that natively contains missing values:

s = pd.Series([1, 2, pd.NA], dtype="Int64")
print(s.dtype)   # Int64  — no upcast to float
print(s)
# 0       1
# 1       2
# 2    <NA>
# dtype: Int64

pd.NA also propagates through boolean operations more predictably than np.nan, following three-valued logic (True, False, NA) rather than always returning NaN.

Removing Missing Values with dropna()

The most direct way to deal with missing data is to remove the rows or columns that contain it. Pandas provides df.dropna() for this purpose. With no arguments, it drops every row that contains at least one missing value:

df_clean = df.dropna()
print(df_clean)
#     name   age  score
# 0  Alice  25.0   88.0

Only row 0 survives because every other row has at least one NaN. The original DataFrame is unchanged; dropna() returns a new object unless you pass inplace=True.

The axis parameter determines the direction of dropping. By default axis=0 drops rows. Setting axis=1 drops columns that contain any missing value, which is useful when an entire column is mostly empty:

df_no_missing_cols = df.dropna(axis=1)
print(df_no_missing_cols)
#     name
# 0  Alice
# 1    Bob
# 2   None
# 3  Diana
# Hmm — only the 'name' column survived (but it also has a None!)
# Let's check more carefully:
print(df.dropna(axis=1))
# Returns only columns where NO value is missing — in this case none survive
# because every column has at least one NaN/None.

The how parameter fine-tunes the dropping condition. how='any' (the default) drops a row if any value in it is missing. how='all' is much more conservative — it drops a row only if every value in that row is missing, which is appropriate when you expect some sparseness but want to eliminate entirely empty records:

df_sparse = pd.DataFrame({
    'a': [1, np.nan, np.nan],
    'b': [2, 3,      np.nan],
    'c': [4, np.nan, np.nan]
})

print(df_sparse.dropna(how='all'))
#      a    b    c
# 0  1.0  2.0  4.0
# 1  NaN  3.0  NaN
# Row 2 is dropped because all its values are NaN.

The thresh parameter offers a middle ground. It keeps only rows that have at least thresh non-NaN values. This is particularly useful for wide datasets where you can tolerate a few missing fields but not rows that are mostly empty:

print(df_sparse.dropna(thresh=2))
#      a    b    c
# 0  1.0  2.0  4.0
# 1  NaN  3.0  NaN
# Row 0 has 3 non-NaN values, row 1 has 1 (fails thresh=2), row 2 has 0.
# Wait — row 1 has only b=3, so 1 non-null value: dropped.
# Only row 0 survives with thresh=2 requiring >=2 non-null values... let's verify:
print(df_sparse.dropna(thresh=2))
#      a    b    c
# 0  1.0  2.0  4.0
# Row 1 has 1 non-null (b), row 2 has 0 non-null — both dropped.

You can also pass a subset parameter listing specific columns to consider when deciding whether to drop a row, which is useful when only certain fields are critical to your analysis.

Imputing Missing Values with fillna()

Dropping rows is simple but comes at the cost of losing potentially valuable data — especially in small datasets. Imputation, the practice of filling missing values with estimated or derived ones, is often preferable. Pandas' df.fillna() method is the primary tool for this.

The most basic form replaces every NaN with a single constant value, which makes sense when you have domain knowledge about what the missing value likely represents:

df_filled = df.fillna(0)
print(df_filled)
#     name   age  score
# 0  Alice  25.0   88.0
# 1    Bob   0.0   91.0
# 2      0  30.0    0.0
# 3  Diana  22.0    0.0

To apply different fill values per column, pass a dictionary mapping column names to their fill values:

df_filled = df.fillna({'name': 'Unknown', 'age': df['age'].median(), 'score': 0})
print(df_filled)
#      name   age  score
# 0   Alice  25.0   88.0
# 1     Bob  25.0   91.0  # median of [25, 30, 22] = 25.0
# 2  Unknown  30.0    0.0
# 3   Diana  22.0    0.0

Forward fill (method='ffill') propagates the last valid observation forward to fill subsequent NaN values. This is especially natural for time-series data where the most recently observed value is a reasonable proxy for missing future readings:

ts = pd.Series([1.0, np.nan, np.nan, 4.0, np.nan])
print(ts.ffill())
# 0    1.0
# 1    1.0
# 2    1.0
# 3    4.0
# 4    4.0

Notice that index 1 and 2 are filled with the previous valid value (1.0), and index 4 is filled with 4.0. If the first value in the Series is NaN, ffill has nothing to propagate and it remains NaN.

Backward fill (method='bfill') works in the opposite direction, using the next valid observation to fill earlier gaps:

print(ts.bfill())
# 0    1.0
# 1    4.0
# 2    4.0
# 3    4.0
# 4    NaN   # no future value to backfill from

Statistical imputation replaces missing values with a computed statistic — most commonly the mean, median, or mode of the column. Mean imputation preserves the column mean but reduces variance and distorts correlations. It is a reasonable first approach when the proportion of missing values is small:

df['score'] = df['score'].fillna(df['score'].mean())
# mean of [88.0, 91.0] = 89.5
print(df['score'])
# 0    88.0
# 1    91.0
# 2    89.5
# 3    89.5

For skewed distributions, median imputation is more robust because the median is not pulled by extreme values. For categorical or discrete variables, mode imputation (filling with the most frequent value) is more appropriate than mean or median.

Interpolation for Missing Values

When data has a natural ordering — such as a time series, a sequence of measurements, or regularly spaced observations — interpolation offers a more sophisticated filling strategy than forward-fill or statistical imputation. Rather than copying a neighboring value or using a global statistic, interpolation estimates missing values by fitting a smooth curve through the surrounding known points.

df.interpolate() with default settings performs linear interpolation, computing a value that lies proportionally between the two nearest non-missing neighbors:

s = pd.Series([0.0, np.nan, np.nan, 9.0, np.nan, 15.0])
print(s.interpolate())
# 0     0.0
# 1     3.0   # 1/3 of the way from 0 to 9
# 2     6.0   # 2/3 of the way from 0 to 9
# 3     9.0
# 4    12.0   # midpoint between 9 and 15
# 5    15.0

The method parameter controls the interpolation algorithm. A few important options include:

  • 'linear' (default): assumes equal spacing between index positions. Does not use the actual index values.
  • 'time': designed for DatetimeIndex data; it uses the actual timestamps to weight the interpolation, so gaps of different durations are handled proportionally.
  • 'polynomial' and 'spline': fit higher-order polynomial curves through the known points, requiring an order argument. Useful for smoothly curved data but can oscillate dangerously if the order is too high or data is sparse.
s_poly = pd.Series([0.0, np.nan, np.nan, 8.0, np.nan, 20.0])
print(s_poly.interpolate(method='polynomial', order=2))

The limit parameter caps how many consecutive NaN values will be filled. This is a safeguard against filling very long gaps where the interpolated values would be unreliable:

s_long_gap = pd.Series([1.0, np.nan, np.nan, np.nan, np.nan, 10.0])
print(s_long_gap.interpolate(limit=2))
# 0     1.0
# 1     2.8   # filled (within limit)
# 2     4.6   # filled (within limit)
# 3     NaN   # not filled (limit=2 exhausted)
# 4     NaN   # not filled
# 5    10.0

Critically, interpolation is only appropriate when the ordering of rows is meaningful. It should never be applied to categorical variables, unordered data, or situations where missingness is not related to position or time. Interpolating a "country" column or a binary flag, for instance, would produce meaningless or invalid values.

Strategy Selection Based on Missingness Type

Choosing the right handling strategy requires understanding why data is missing, not just where. Statisticians classify missingness into three structural types, each with different implications:

Missing Completely at Random (MCAR) means the probability that a value is missing is entirely unrelated to the data itself — observed or unobserved. A sensor randomly failing once a week regardless of the measurement value is an example. MCAR is the most benign case: dropping rows introduces no systematic bias, and simple imputation strategies (mean, median, constant fill) are generally safe, though they still reduce variance and inflate apparent precision.

Missing at Random (MAR) means the probability of missingness is related to other observed variables, but not to the missing value itself. For example, younger survey respondents might be less likely to report income, but among respondents of the same age, income reporting is random. You cannot safely drop MAR data without introducing bias. More advanced imputation methods — such as using a regression model trained on other columns to predict the missing values, or multiple imputation techniques — can recover reliable estimates because the pattern is explainable by observed data.

Missing Not at Random (MNAR) is the most difficult case. Here, the probability of missingness depends on the unobserved value itself. High-income earners might deliberately skip an income question precisely because their income is high. No purely data-driven imputation can fully correct for MNAR because the information you need to impute the missing value is the same information that is missing. Handling MNAR typically requires domain knowledge, sensitivity analysis, or collecting additional data to understand the missingness mechanism.

The following table summarizes the three types and their recommended handling approaches:

Missingness Type Definition Example Recommended Strategy
MCAR Missingness is unrelated to any data Random sensor dropout Drop rows or simple imputation (mean/median/mode)
MAR Missingness depends on other observed variables Age predicts whether income is reported Model-based or multiple imputation using other columns
MNAR Missingness depends on the missing value itself High earners skip income question Domain-specific treatment, sensitivity analysis, data collection

Beyond structural type, the proportion of missing data matters. As a rough guide: columns with more than 40–50% missing values are often better dropped outright, as any imputation would be fabricating more than half the data. Columns with under 5% missing data are usually safe to impute or drop rows with minimal impact on the analysis.

Visualizing missingness patterns can reveal relationships between missing fields that are invisible from column-level counts alone. The missingno library offers several useful plots for this. msno.matrix(df) renders a visual grid where present values appear as white bars and missing values appear as dark gaps, allowing you to spot row-level correlations — for example, whether the same rows tend to be missing across multiple columns simultaneously. msno.heatmap(df) shows the correlation between columns' missingness patterns numerically. msno.bar(df) produces a simple bar chart of non-null counts per column.

# pip install missingno
import missingno as msno
import matplotlib.pyplot as plt

msno.matrix(df)
plt.show()

msno.heatmap(df)
plt.show()

A strong positive correlation in the heatmap between two columns' missingness patterns (meaning they tend to be missing in the same rows) is a signal that the data may be MAR — the missingness of one might be predictable from the other — which in turn informs a more sophisticated imputation strategy rather than simple row deletion.

In summary, handling missing data well is not a single-step mechanical operation but an analytical process: detect the scope of the problem, understand the type of missingness, choose a strategy proportionate to the extent and mechanism of the gaps, apply it carefully, and validate that the resulting dataset behaves as expected before moving on to analysis or modeling.
