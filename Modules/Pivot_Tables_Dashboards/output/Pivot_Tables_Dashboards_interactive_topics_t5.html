Introduction to Power Query

▶

Data rarely arrives in the exact shape an analyst needs. Files come from different systems, columns are misnamed, dates are formatted inconsistently, and rows contain errors that must be removed before any meaningful analysis can begin. Traditionally, Excel users handled all of this by hand — copying and pasting, writing formulas, or recording fragile macros. Power Query fundamentally changes that workflow. It is a built-in data connectivity and transformation engine available in Excel (and Power BI) that lets you pull data in from virtually any source, clean and reshape it using a point-and-click interface, and then reload the finished result into your workbook automatically every time the underlying data changes. Understanding what Power Query is, where it gets its data, and how its editor and refresh system work gives you the foundation to build data workflows that are fast, reliable, and repeatable.

What Is Power Query?

Power Query is best understood through the lens of a concept borrowed from data engineering: ETL, which stands for Extract, Transform, Load. Each word describes a distinct stage of the data preparation journey.

  • Extract — Power Query reaches out to a data source (a CSV file, a database, a website, a cloud service) and pulls the raw data into a staging environment where you can inspect it.
  • Transform — Inside the Power Query Editor, you apply a series of cleaning and shaping operations: removing blank rows, splitting columns, changing data types, filtering out irrelevant records, merging tables, and much more.
  • Load — Once the data looks exactly the way you need it, Power Query pushes the finished table into an Excel worksheet or directly into the workbook's internal data model, ready for PivotTables, charts, or formulas.

What makes Power Query special compared to doing this work manually is its step recording capability. Every action you take inside the editor — renaming a column, removing duplicates, filtering a date — is recorded as a discrete step written in Power Query's own formula language, called M. These steps are stored in order and re-executed automatically every time you refresh the query. You never have to redo your data cleaning by hand. If a new month's data arrives in the same CSV file tomorrow, one click replays every transformation against the new data and delivers a freshly cleaned result in seconds.

This design also dramatically reduces human error. Manual data preparation invites mistakes: a row accidentally deleted, a formula dragged one cell too far, a filter left on. Because Power Query's transformations are scripted and consistent, the same logic runs the same way every time, on every refresh, without variation.

Supported Data Sources

One of Power Query's greatest strengths is the breadth of systems it can connect to. Rather than forcing you to export everything to a CSV first, it speaks the native language of dozens of source types. Sources broadly fall into three families:

  • Flat files and local workbooks — This is the most common starting point for Excel users. Power Query can import comma-separated values (.csv), plain text files (.txt) with any delimiter, and other Excel workbooks (.xlsx, .xlsb). It can even loop through an entire folder of similarly structured files and stack them into one unified table automatically — a task that would take hours to do manually.
  • Relational databases — Power Query includes native connectors for Microsoft SQL Server, Microsoft Access, MySQL, PostgreSQL, Oracle, and others. You provide the server address, database name, and credentials, and Power Query lets you browse tables and views just as a database tool would. For large databases, Power Query can even push filter and aggregation logic back to the database server (a feature called query folding), so only the rows you actually need travel across the network.
  • Web-based and cloud sources — Power Query can scrape structured tables directly from web pages, connect to SharePoint lists and document libraries, pull from Microsoft Dataverse, import JSON or XML feeds from REST APIs, and connect to cloud platforms such as Azure, Salesforce, and Google Analytics through dedicated connectors. This makes it possible to build live-updating Excel reports driven entirely by cloud data without ever manually downloading a file.

The table below summarises some representative source types and their typical use cases:

Connecting Excel to an External Dataset

The process of establishing a new Power Query connection follows a consistent pattern regardless of the source type. Here is how it works in practice:

  • Step 1 — Open the Get Data menu. On the Excel ribbon, click the Data tab. In the Get & Transform Data group you will find the Get Data button. Clicking it reveals a categorised menu of all available source types — From File, From Database, From Azure, From Online Services, and so on. Selecting the appropriate category then presents the specific connector options within it.
  • Step 2 — Provide connection details. For a CSV file, you simply browse to the file location. For a SQL Server database, you type the server address, choose the database, and (if required) enter credentials. Power Query stores these details securely as part of the workbook's connection definitions.
  • Step 3 — Preview before committing. Before anything is loaded into your spreadsheet, Power Query displays a Navigator window or a data preview pane. This is your chance to verify that you have connected to the right table or file and that the data looks as expected. You can see column names, data types, and sample rows. From here you choose either Load (send the data straight to Excel as-is) or Transform Data (open the Power Query Editor to clean the data first — the recommended path for any real-world dataset).
  • Step 4 — Connection details are saved. Once you have completed the setup, the connection is stored inside the workbook. Future refreshes do not require you to re-enter the file path or server address. Excel remembers everything and uses those saved details to re-run the query on demand.

For example, suppose your company's finance team drops a new CSV file called sales_data.csv into a shared network folder at the end of each month. You would connect to that file once using Get Data → From File → From Text/CSV, apply your cleaning steps in the editor, and load the result. Next month, when the file is updated, you simply click Refresh and the new data flows through the same pipeline automatically.

The Power Query Editor

When you choose Transform Data after connecting to a source, the Power Query Editor opens as a separate window. It is the core workspace where all data shaping happens, and it is worth understanding its layout in detail.

  • The data preview grid — The central area shows a spreadsheet-like grid of your data. Column headers sit at the top, and you can scroll through rows to inspect values. Unlike a regular Excel sheet, this grid is read-only; you cannot type into cells directly. Instead, all changes are made through menu commands, right-click options, or the formula bar, which keeps the transformation logic cleanly separated from the data itself.
  • The Queries pane (left side) — If your workbook contains multiple queries, they are listed here. You can click between them to switch the editor's view to a different query. This makes it easy to manage a complex workbook that pulls from several different sources.
  • The Applied Steps pane (right side) — This is the heart of Power Query's power. Every action you take is recorded here as a named step. Common automatic steps include Source (the connection itself), Navigation (selecting a specific table or sheet), and Changed Type (Power Query's automatic guess at column data types). Steps you add manually — such as Removed Columns, Filtered Rows, or Renamed Columns — appear beneath these. You can click any step to see what the data looked like at that point in the sequence, rename steps to make them self-documenting, delete steps you no longer need, or reorder them by dragging. This creates a full, inspectable audit trail of every decision made during data preparation.
  • The ribbon — The editor has its own multi-tab ribbon with commands organised into groups: Home (load, close, refresh), Transform (reshape existing columns), Add Column (create new calculated columns), and View (formula bar, data profiling tools). Most common operations — changing a data type, splitting a column by delimiter, replacing values — are a single click or right-click away.

As a concrete example, imagine you import a CSV of customer orders. Power Query automatically detects columns for OrderDate, CustomerName, Product, Quantity, and Price. You notice that OrderDate has been read as plain text rather than a date. You click the column header's type icon and select Date — a Changed Type step is recorded. You then right-click the CustomerName header and choose Transform → Trim to remove accidental leading and trailing spaces — a Trimmed Text step is added. You filter the Product column to exclude a discontinued item — a Filtered Rows step appears. The Applied Steps pane now reads like a clear recipe: anyone reviewing it can understand exactly what was done to the raw data and why.

Automating Data Import with Refresh

The refresh mechanism is what converts Power Query from a one-time import tool into a genuine automation platform. When you click Refresh (either by right-clicking the query in the Queries & Connections pane, or using Data → Refresh All on the ribbon), Excel re-executes every step in the Applied Steps sequence from scratch against the current state of the data source. The raw data is re-extracted, all transformations are re-applied in order, and the finished table in your worksheet is replaced with the updated result.

  • On-demand refresh — You control exactly when the data updates by clicking Refresh manually. This is appropriate when data sources update infrequently or when you want to review results before they appear in reports.
  • Refresh on open — In the query's Properties settings (accessible via Data → Queries & Connections → right-click a query → Properties), you can enable the option Refresh data when opening the file. With this setting active, every time anyone opens the workbook, Power Query silently re-runs all queries and delivers the latest data before the user sees the sheet. Dashboards built on this data are always current.
  • Scheduled refresh via Power BI or SharePoint — If you publish the workbook to SharePoint or Power BI, you can schedule automated refreshes on a timed basis (for example, every morning at 6 a.m.) without anyone needing to open the file at all.

Consider a practical scenario: a sales manager has an Excel dashboard with a PivotTable summarising regional revenue. The underlying data comes from a Power Query that connects to a SQL Server database. Every Monday morning when the manager opens the workbook, the query refreshes automatically, pulling the latest week's transactions, applying all the cleaning steps, and updating the PivotTable — all before the manager has even looked at the numbers. What used to take 30 minutes of manual export and formatting now happens invisibly in seconds.

Why Power Query Improves Data Workflows

The cumulative effect of Power Query's features represents a genuine shift in how Excel can be used for data work. Several benefits stand out as particularly important:

  • Reusability eliminates repeated effort. The single biggest time cost in traditional spreadsheet work is redoing the same data cleaning steps every time new data arrives. Power Query abolishes that cost. You do the work once, record the steps, and the engine repeats them indefinitely. An organisation that previously spent hours each month reformatting a supplier's data export can reduce that to a one-time setup and zero ongoing effort.
  • Multiple sources, one destination. Real analytical questions rarely have a single data source. You might need to combine sales transactions from a database with customer demographic data from a CSV and product categories from a SharePoint list. Power Query can connect to all three simultaneously, merge the tables using common key columns (a process called merging queries), and deliver a single, unified table. This kind of multi-source integration used to require specialised database skills; Power Query makes it accessible to any Excel user.
  • Transparency and auditability. Because every transformation is recorded as a named, inspectable step, Power Query creates an automatic audit trail. A colleague reviewing your workbook can open the editor and follow the Applied Steps to understand exactly how the raw data was turned into the finished report. This is invaluable for compliance, peer review, and troubleshooting. If a number in a report looks wrong, you can trace back through each step to find exactly where a value changed and why — something that is essentially impossible with manual copy-paste workflows.
  • Reduced human error. Manual data preparation is error-prone by nature. Power Query's deterministic, scripted approach means the same logic is applied consistently every single time. Once a transformation is correct, it stays correct across every future refresh.
  • Scalability. Whether your dataset has 100 rows or 10 million, Power Query handles the transformation pipeline the same way. For very large datasets, features like query folding push processing to the database server rather than Excel's memory, keeping performance manageable even at scale.

In short, Power Query does not just make importing data easier — it reimagines the entire relationship between Excel and external data. It turns Excel from a destination where data is pasted and manually groomed into a live, connected reporting environment where clean, current data arrives automatically and the analyst's time is freed for actual analysis rather than data wrangling.

NotesConsider demonstrating the Applied Steps pane live during instruction — seeing steps appear in real time as transformations are applied is one of the most effective ways to make the concept concrete for new learners. Emphasise that students should always choose "Transform Data" rather than "Load" when first connecting to a new source, so they build the habit of inspecting and cleaning data before it enters the worksheet.
