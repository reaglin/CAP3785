Aggregation with GROUP BY and HAVING

▶

One of the most powerful capabilities in SQL is the ability to summarize and analyze data across many rows at once. Rather than returning every individual record, you can instruct the database engine to compute totals, averages, counts, and other statistics — either across the entire table or within logical groupings of rows. This is the domain of aggregate functions, the GROUP BY clause, and the HAVING clause. Together, these tools transform raw relational data into meaningful summaries that drive reporting, dashboards, and analytical queries of every kind.

Understanding aggregation well requires appreciating the order in which SQL processes a query. Conceptually, the database engine works through the clauses in this sequence: FROM (identify the source tables) → WHERE (filter individual rows) → GROUP BY (collapse rows into groups) → aggregate functions (compute per-group statistics) → HAVING (filter groups) → SELECT (project columns) → ORDER BY (sort) → LIMIT/TOP (restrict output). This logical order explains why WHERE cannot reference aggregates and why HAVING can — a distinction that trips up many learners.

Introduction to Aggregate Functions

Aggregate functions take a set of rows as input and return a single scalar value. The most commonly used aggregate functions in SQL are COUNT, SUM, AVG, MIN, and MAX. When used in a query without any GROUP BY clause, they collapse the entire result set into one row.

COUNT comes in two important flavors that behave very differently:

  • COUNT(*) counts every row in the group, including rows that contain NULL values in any column. It simply asks: how many rows exist here?
  • COUNT(column_name) counts only rows where that specific column is not NULL. If a column has 200 rows but 40 of them contain NULL, COUNT(column_name) returns 160.

Consider a table called orders with columns order_id, customer_id, amount, and discount_code (which is NULL when no discount was applied). The difference in behavior is illustrated here:

-- Counts every row in the table
SELECT COUNT(*) AS total_orders
FROM orders;

-- Counts only rows where discount_code is not NULL
SELECT COUNT(discount_code) AS discounted_orders
FROM orders;

If the table has 500 rows and only 120 have a discount code, the first query returns 500 and the second returns 120.

SUM adds up all non-NULL values in a numeric column. If every value in the column is NULL, SUM returns NULL (not zero). AVG computes the arithmetic mean of all non-NULL values — it divides the sum by the count of non-NULL rows, not the total row count. This means NULLs are excluded from the average calculation, which can produce surprising results if NULLs are conceptually meant to represent zero.

SELECT
    SUM(amount)   AS total_revenue,
    AVG(amount)   AS average_order_value
FROM orders;

MIN and MAX work across any orderable data type — numbers, dates, and even strings (using lexicographic ordering). They return the smallest and largest non-NULL values respectively.

SELECT
    MIN(order_date) AS first_order_date,
    MAX(order_date) AS most_recent_order_date,
    MIN(amount)     AS smallest_order,
    MAX(amount)     AS largest_order
FROM orders;

When used without GROUP BY, all five of these functions reduce the entire table to a single summary row. That is useful for quick statistics, but the real analytical power emerges when you combine them with GROUP BY.

Grouping Data with GROUP BY

The GROUP BY clause instructs the database to partition the rows of the result set into distinct groups based on the values in one or more columns. Each unique combination of values in the GROUP BY columns becomes exactly one output row, with aggregate functions applied independently within each group.

Imagine a sales table storing individual transaction records:

sale_id region salesperson product amount
1NorthAliceWidget200
2NorthBobGadget150
3SouthCarolWidget300
4SouthCarolGadget100
5NorthAliceGadget250
6SouthBobWidget180

Grouping by region collapses all rows sharing the same region value into one summary row:

SELECT
    region,
    COUNT(*)      AS num_sales,
    SUM(amount)   AS total_sales
FROM sales
GROUP BY region;

This produces:

region num_sales total_sales
North3600
South3580

You can group by more than one column, which creates a sub-group for every unique combination of the specified column values. Grouping the same table by both region and salesperson yields a finer-grained breakdown:

SELECT
    region,
    salesperson,
    COUNT(*)    AS num_sales,
    SUM(amount) AS total_sales
FROM sales
GROUP BY region, salesperson;

Result:

region salesperson num_sales total_sales
NorthAlice2450
NorthBob1150
SouthBob1180
SouthCarol2400

A critical rule: every column in the SELECT clause that is not inside an aggregate function must appear in the GROUP BY clause. If you try to SELECT salesperson without including it in GROUP BY, the database cannot know which salesperson's name to display for a group that may contain multiple salespeople — this is an ambiguity error in strict SQL engines (MySQL in certain modes is a notable exception, though its behavior is unreliable).

Also important: the output order of GROUP BY results is not guaranteed. Without an explicit ORDER BY, different database engines may return grouped rows in any order, and that order can change between executions. Always add ORDER BY if you need a predictable sequence.

The concept of cardinality matters for understanding GROUP BY output volume. A low-cardinality column like region (only a handful of distinct values) produces a compact summary with few rows. A high-cardinality column like customer_id (potentially millions of distinct values) produces many output rows — one per customer — offering a detailed but voluminous breakdown.

Combining Aggregate Functions with GROUP BY

A single SELECT statement can contain multiple aggregate functions applied simultaneously to the same groups, which allows rich, multi-dimensional summaries in one query. Combine this with column aliases and arithmetic expressions inside aggregates for highly readable, expressive results.

SELECT
    region,
    salesperson,
    COUNT(*)                            AS num_sales,
    SUM(amount)                         AS total_revenue,
    AVG(amount)                         AS avg_sale_value,
    MIN(amount)                         AS min_sale,
    MAX(amount)                         AS max_sale
FROM sales
GROUP BY region, salesperson
ORDER BY region, total_revenue DESC;

Aliases (introduced with the AS keyword) are essential for readability. Column headers like COUNT(*) or SUM(amount) are awkward in reports and application code; names like num_sales and total_revenue communicate intent clearly.

Aggregate functions can also wrap arithmetic expressions. Suppose a table called order_items has columns quantity and unit_price. You can compute revenue per product directly inside the aggregate:

SELECT
    product_id,
    SUM(quantity * unit_price)          AS total_revenue,
    SUM(quantity)                       AS units_sold,
    AVG(unit_price)                     AS avg_unit_price
FROM order_items
GROUP BY product_id
ORDER BY total_revenue DESC;

Here SUM(quantity * unit_price) multiplies the two column values for each row first, then sums those products across all rows in the group. This is different from — and generally more useful than — SUM(quantity) * AVG(unit_price), which would produce a different (mathematically incorrect) result.

Filtering Grouped Results with HAVING

Once rows have been grouped and aggregated, you often want to show only the groups that meet some criterion — for example, only regions with more than ten sales, or only products whose total revenue exceeds $50,000. This is exactly what HAVING is for.

HAVING is evaluated after GROUP BY and aggregation, which means it can reference aggregate function results directly. WHERE, by contrast, is evaluated before grouping and cannot reference aggregates at all — attempting to write WHERE COUNT(*) > 5 will produce an error in every standard SQL engine.

-- Show only salesperson-region combinations with total sales above 300
SELECT
    region,
    salesperson,
    SUM(amount) AS total_revenue
FROM sales
GROUP BY region, salesperson
HAVING SUM(amount) > 300
ORDER BY total_revenue DESC;

Result from the earlier dataset:

region salesperson total_revenue
NorthAlice450
SouthCarol400

North/Bob (150) and South/Bob (180) are excluded because their totals do not exceed 300.

Multiple conditions can be combined in HAVING using AND and OR:

SELECT
    product_id,
    COUNT(*)            AS num_orders,
    SUM(amount)         AS total_revenue
FROM order_items
GROUP BY product_id
HAVING COUNT(*) >= 10
   AND SUM(amount) > 5000;

This returns only products that appear in at least 10 orders and whose total revenue exceeds 5,000.

Regarding aliases in HAVING: the SQL standard requires the full aggregate expression in HAVING rather than the alias defined in SELECT, because SELECT is logically processed after HAVING. Most strict engines (PostgreSQL, SQL Server, Oracle) enforce this. Writing HAVING total_revenue > 5000 may work in MySQL and SQLite as a convenience extension, but for portability you should write HAVING SUM(amount) > 5000.

An unusual but valid edge case: HAVING without GROUP BY. In this situation, the entire result set is treated as a single group, and HAVING filters that one group. This is rarely written intentionally but is sometimes useful for existence checks:

-- Returns one row only if the table has more than 1000 rows total
SELECT COUNT(*) AS total_rows
FROM orders
HAVING COUNT(*) > 1000;

WHERE vs. HAVING: Knowing When to Use Each

The distinction between WHERE and HAVING is one of the most commonly misunderstood aspects of SQL aggregation. The rule is straightforward once you internalize the logical processing order:

  • WHERE filters individual rows from the base table(s) before any grouping or aggregation takes place. It cannot reference aggregate functions.
  • HAVING filters groups after GROUP BY and aggregation. It can — and typically does — reference aggregate functions.

A query can contain both clauses, applying filtering at two separate stages:

SELECT
    region,
    salesperson,
    COUNT(*)    AS num_sales,
    SUM(amount) AS total_revenue
FROM sales
WHERE amount > 100                   -- pre-filter: exclude individual sales of 100 or less
GROUP BY region, salesperson
HAVING SUM(amount) > 300             -- post-filter: exclude groups with total <= 300
ORDER BY total_revenue DESC;

The WHERE clause here removes the South/Bob/Widget row (amount = 180 — wait, 180 > 100, so it stays; imagine if that row had amount = 80) before any grouping happens. The HAVING clause then removes any salesperson-region groups whose combined total is 300 or less after aggregation.

Using WHERE to pre-filter before aggregation is not just logically correct — it is an important performance optimization. Filtering with WHERE reduces the number of rows that the database engine must process during the GROUP BY and aggregation phases. If you have a sales table with 10 million rows but are only interested in sales from the current year, placing WHERE YEAR(sale_date) = 2024 in your query means the engine aggregates perhaps 800,000 rows rather than all 10 million. This can make a dramatic difference in execution time, especially if the WHERE column is indexed.

A common mistake is trying to use WHERE where HAVING is needed:

-- WRONG: Cannot use aggregate in WHERE
SELECT region, SUM(amount) AS total
FROM sales
WHERE SUM(amount) > 500    -- ERROR
GROUP BY region;

-- CORRECT: Use HAVING for aggregate conditions
SELECT region, SUM(amount) AS total
FROM sales
GROUP BY region
HAVING SUM(amount) > 500;

Conversely, filtering a plain (non-aggregated) column condition in HAVING instead of WHERE is legal but wasteful:

-- Legal but inefficient: this should be in WHERE
SELECT region, SUM(amount) AS total
FROM sales
GROUP BY region
HAVING region = 'North';   -- works, but forces all regions to be aggregated first

-- Better: filter before grouping
SELECT region, SUM(amount) AS total
FROM sales
WHERE region = 'North'
GROUP BY region;

Ordering and Presenting Aggregated Results

ORDER BY is applied as the last logical step before the result set is returned to the client (and before any LIMIT/TOP restriction is applied). Because it runs after SELECT, ORDER BY can freely reference column aliases defined in the SELECT clause, which is a convenience most SQL engines support.

SELECT
    department,
    COUNT(*)        AS headcount,
    AVG(salary)     AS avg_salary
FROM employees
GROUP BY department
ORDER BY avg_salary DESC;    -- ordering by alias: supported in most engines

You can also order by the full aggregate expression, which is unambiguous across all SQL dialects:

ORDER BY AVG(salary) DESC

Sorting aggregated results in descending order is an extremely common pattern for "top-N" reporting. Combined with LIMIT (or TOP in SQL Server / T-SQL), this enables queries that surface the highest-performing groups:

-- Top 5 products by total revenue
SELECT
    product_id,
    SUM(quantity * unit_price) AS total_revenue
FROM order_items
GROUP BY product_id
ORDER BY total_revenue DESC
LIMIT 5;

In SQL Server the equivalent syntax uses TOP in the SELECT clause:

SELECT TOP 5
    product_id,
    SUM(quantity * unit_price) AS total_revenue
FROM order_items
GROUP BY product_id
ORDER BY total_revenue DESC;

These top-N aggregation queries are the backbone of leaderboard reports, sales rankings, inventory analyses, and countless other business intelligence use cases.

A complete, realistic example pulling all concepts together: find the top 3 salespeople by total revenue in the current year, considering only transactions above $50, and only including salespeople who have made at least 5 qualifying sales:

SELECT
    salesperson,
    COUNT(*)            AS qualifying_sales,
    SUM(amount)         AS total_revenue,
    AVG(amount)         AS avg_sale_value
FROM sales
WHERE amount > 50
  AND YEAR(sale_date) = 2024
GROUP BY salesperson
HAVING COUNT(*) >= 5
ORDER BY total_revenue DESC
LIMIT 3;

Walking through the execution:

  • FROM sales: start with the full sales table.
  • WHERE: keep only rows where amount > 50 and the sale occurred in 2024.
  • GROUP BY salesperson: collapse remaining rows into one group per salesperson.
  • Aggregate functions: compute COUNT, SUM, and AVG per group.
  • HAVING: discard groups where the salesperson has fewer than 5 qualifying sales.
  • SELECT: project the four named columns.
  • ORDER BY total_revenue DESC: rank remaining groups by descending revenue.
  • LIMIT 3: return only the top three rows.

Mastering this full pipeline — aggregate functions, GROUP BY, HAVING, WHERE, and ORDER BY working together — gives you the tools to answer virtually any summarization question against a relational database.
