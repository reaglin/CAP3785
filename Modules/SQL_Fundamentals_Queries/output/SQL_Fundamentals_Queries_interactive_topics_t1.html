Introduction to SQL and Relational Databases

▶

SQL, which stands for Structured Query Language, is the standard language used to communicate with relational databases. It was first developed in the 1970s at IBM and has since become one of the most widely used tools in data management. Whether you are retrieving a customer's order history, inserting a new product into a catalog, or deleting outdated records, SQL provides a consistent, powerful syntax for all of these operations. Almost every major technology company, government agency, e-commerce platform, and analytics team relies on SQL in some form to store and work with data.

SQL allows users to create, read, update, and delete data stored in a database — a set of operations commonly abbreviated as CRUD. These four actions form the backbone of nearly every interaction with a database system:

  • Create: Add new data to the database, such as inserting a new customer record or creating an entirely new table.
  • Read: Retrieve existing data using queries, such as fetching all orders placed in the last 30 days.
  • Update: Modify existing records, such as changing a user's email address or updating a product price.
  • Delete: Remove records from the database, such as purging expired session tokens or archiving old transactions.

One of the most important characteristics of SQL is that it is a declarative language. This means that when you write a SQL query, you describe what result you want — not how the computer should go about finding it. For example, if you want a list of all customers from New York, you simply state that intention in SQL. The database engine then figures out the most efficient way to scan, filter, and return that data on your behalf. This is in contrast to procedural languages like Python or Java, where you would need to write step-by-step instructions for looping through records and checking each one manually. The declarative nature of SQL makes it accessible to analysts and business users, not just professional programmers.

SQL is used across a wide range of roles and industries. Data analysts use it to answer business questions like "Which product had the highest revenue last quarter?" Software developers use it to build applications that store and retrieve user data. Data scientists use it to extract and prepare datasets for machine learning models. Even product managers and marketing teams often learn basic SQL to pull their own reports without relying on a technical team.

There are several popular SQL database systems in widespread use today. While they all speak the SQL language, each system has its own extensions, performance characteristics, and use cases:

Database System Common Use Cases Notable Traits
MySQL Web applications, e-commerce platforms Open source, widely used with PHP and WordPress
PostgreSQL Enterprise applications, analytics Open source, highly standards-compliant, supports advanced data types
SQLite Mobile apps, embedded systems, local development Serverless, stored as a single file, extremely lightweight
Microsoft SQL Server Enterprise business applications, Microsoft ecosystems Deep integration with Microsoft tools, powerful reporting features

Despite their differences, the core SQL syntax you learn — such as SELECT, WHERE, JOIN, and GROUP BY — is transferable across all of these systems. Learning SQL once gives you a foundation applicable almost everywhere data is stored.

To understand SQL fully, you first need to understand the system it was built to work with: the relational database. A relational database organizes data into one or more tables, where each table represents a distinct real-world entity or concept. For example, a retail business might have a table for customers, another for products, and another for orders. Each of these tables captures specific information about that entity in a structured, consistent format.

The word "relational" refers to the ability to link tables together using keys — forming relationships between entities. For instance, an order belongs to a customer and contains products. Rather than duplicating all customer information inside the orders table (which would be wasteful and error-prone), relational databases store each piece of information once and use references to connect related data. This design principle directly supports two major goals:

  • Reduced data redundancy: Information is stored in one place and referenced elsewhere, so there is no need to repeat the same customer name and address across hundreds of order records.
  • Improved data integrity: When data lives in one authoritative location, updates are simpler and the risk of contradictory or stale data is minimized. If a customer changes their address, updating it in the customers table automatically reflects that change everywhere it is referenced.

The table is the core structural unit of a relational database. Every table is given a unique name that clearly reflects the type of data it contains. Well-chosen table names like employees, sales, products, or invoices make the database self-documenting and easier to navigate. All data stored within a single table follows the same structure, which is defined upfront by its columns. A database for a complete application typically contains many tables, and together these tables form the full data model — a map of all the information the system needs to function.

Think of a database table as very similar in appearance to a spreadsheet — it has named columns across the top and rows of data running downward. But unlike a spreadsheet, a database table enforces strict rules about what data can be stored, making it far more reliable for large-scale data management.

Let us look at a concrete example. Below is a simplified representation of a customers table:

customer_id first_name last_name email city
1 Alice Johnson alice@example.com New York
2 Bob Martinez bob@example.com Chicago
3 Clara Lee clara@example.com Los Angeles

In this table, each column defines a specific attribute of the entity being stored. The first_name column stores text representing a person's given name. The city column stores text representing the city where the customer lives. Every column has a defined data type that controls what kind of values are allowed in it. Common data types include:

  • INTEGER or INT: Whole numbers, such as 1, 42, or 1000. Used for counts, IDs, and quantities.
  • VARCHAR or TEXT: Variable-length text strings, such as names, addresses, and email addresses.
  • DATE or DATETIME: Calendar dates and timestamps, such as a customer's sign-up date or an order's placement time.
  • DECIMAL or FLOAT: Numbers with decimal points, such as prices, measurements, or percentages.
  • BOOLEAN: True or false values, such as whether a user's account is active.

Data types serve as a form of built-in validation. If a column is defined as an INTEGER, the database will refuse to store the word "hello" in it. This catches errors at the point of entry and ensures that data remains consistent and usable for calculations, sorting, and filtering.

Each row in a table represents one complete, individual record. In the customers table above, each row is one customer. In a sales table, each row might be one transaction. In a products table, each row is one product. Every row in a given table shares exactly the same columns, so all records are structurally uniform. You will never find a customer row that has a price column, because that column simply does not belong to the customers table — it belongs to the products table. This strict structure is what makes databases so reliable and queryable at scale.

A critical concept in relational databases is the primary key. Every table should designate one column (or occasionally a combination of columns) as its primary key. The primary key is the unique identifier for each row in the table. Looking back at the customers table, the customer_id column serves as the primary key — each customer has a distinct ID number that no other customer shares.

Primary keys must satisfy two strict requirements:

  • Uniqueness: No two rows in the table can have the same primary key value. There cannot be two customers both assigned customer_id = 1.
  • Non-null: A primary key value can never be empty or missing. Every record must have an identifier.

Primary keys are foundational to data integrity and to the relational model itself. When you want to link tables together — for example, connecting each order to the customer who placed it — you store the customer's primary key (customer_id) inside the orders table. This reference is called a foreign key, and it is how relationships between tables are established. Without reliable primary keys, you cannot build trustworthy relationships, and the relational model breaks down.

In practice, primary keys are often auto-generated integers that increment with each new record (1, 2, 3, and so on), though they can also be natural identifiers like email addresses or product codes when those are guaranteed to be unique.

Now that you understand the structure of a relational database, you can appreciate how SQL interacts with it. Every SQL query targets a specific table by name and specifies what data should be retrieved or manipulated. The most fundamental SQL statement is SELECT, which reads data from a table. Here is a simple example:

SELECT first_name, last_name, city
FROM customers;

This query instructs the database to go to the customers table and return the first_name, last_name, and city columns for every row. The result would look exactly like a subset of the table shown earlier. Notice that you did not instruct the database how to find these records — you simply declared what you wanted, and the database engine handled the rest.

You can also filter rows by adding conditions. For example, to retrieve only customers who live in Chicago:

SELECT first_name, last_name, email
FROM customers
WHERE city = 'Chicago';

This query returns only the row for Bob Martinez, because he is the only customer in Chicago. SQL's ability to filter, sort, aggregate, and join data across multiple tables is what makes it such a powerful tool for answering real business questions.

Before writing any query, it is essential to understand the structure of the table you are querying — specifically, the column names and their data types. Writing WHERE city = 'Chicago' only works because you know there is a column called city that stores text values. Trying to filter on a column that does not exist, or comparing a text column to a number, will produce an error or unexpected results. This is why understanding the data model — the tables, columns, keys, and relationships — is always the first step before writing SQL queries.

SQL commands are executed against a live database in real time. Unlike a static spreadsheet file, a database is continuously updated as new transactions occur, new users register, or new products are added. This means a SQL query run today may return different results than the same query run last week, because the underlying data has changed. This real-time nature is one of SQL's greatest strengths — it allows you to always work with the most current information available, making it invaluable for business reporting, operational monitoring, and data-driven decision-making.
