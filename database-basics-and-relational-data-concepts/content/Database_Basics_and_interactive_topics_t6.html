Relationships Between Tables

▶

At the heart of every relational database is a deceptively simple idea: instead of cramming all information into one enormous table, you break it into smaller, focused tables and then define the relationships between them. Those relationships are what give the relational model its name, its power, and its elegance. Understanding how tables relate to one another is therefore not a peripheral concept — it is the foundation upon which every meaningful database is built.

A table relationship is a logical connection between two tables, established through shared key values. One table exposes a primary key — a column (or combination of columns) whose values uniquely identify every row — and another table references that key through a foreign key column. When a query needs information from both tables, it can follow that link and combine the rows, a process called a join. Without relationships, every piece of information that belongs together would have to be repeated in full wherever it was needed, leading to bloated, inconsistent, and difficult-to-maintain data.

Consider a simple business scenario. A company tracks customers and the orders those customers place. Without relationships, every order record would have to store the customer's full name, address, phone number, and email address. The moment a customer moves house, every one of their order rows must be updated. Miss even one and the database contains contradictory facts. With a relationship, the customer's details live in a customers table exactly once. Each order row in an orders table stores only a small customer_id value that points back to the one true record. Update the address in one place and every order automatically reflects the change. This reduction of redundancy — and the consistency it enforces — is the first and most practical benefit of table relationships.

Relationships also enable the database engine to enforce referential integrity: the guarantee that a foreign key value will always point to a row that actually exists. If orders.customer_id = 42, then a row with customers.customer_id = 42 must exist. The database can reject any attempt to insert an orphan order or to delete a customer who still has outstanding orders, depending on the rules you configure. This kind of automated data quality checking would be impossible to replicate reliably in application code alone.

There are three classic patterns of relationship, each describing a different cardinality — that is, how many rows on one side of the relationship can be associated with how many rows on the other.

One-to-One Relationships

In a one-to-one relationship, each row in Table A corresponds to at most one row in Table B, and vice versa. This is the rarest of the three patterns because, at first glance, it seems like the two tables could simply be merged into one. The split is nevertheless useful in several practical situations.

Security is a common motivation. Imagine an employees table that all managers can query, and a separate employee_salary_details table that only the payroll team may access. Keeping sensitive columns in their own table makes it straightforward to grant or restrict permissions at the table level without exposing everything to everyone.

Performance is another reason. A products table might be queried millions of times per day just to display a product's name and price. Storing large binary objects — high-resolution images, detailed specification documents — in those same rows would make every routine query slower because the database engine must handle far wider rows. Splitting the large data into a product_details table means the main table stays lean and fast, while the bulky data is fetched only when truly needed.

Optional extension is a third use case. Not every employee has a company vehicle, so rather than adding nullable vehicle columns to the main employees table, you create a separate employee_vehicles table that simply has no row for employees without a vehicle.

Mechanically, a one-to-one relationship is implemented by placing the foreign key in one of the two tables and adding a UNIQUE constraint to that foreign key column. The unique constraint is what makes it one-to-one rather than one-to-many: it prevents a second row from claiming the same parent record.

-- Parent table
CREATE TABLE employees (
    employee_id   INT          PRIMARY KEY,
    full_name     VARCHAR(100) NOT NULL,
    department_id INT
);

-- Extension table — one optional row per employee
CREATE TABLE employee_salary_details (
    employee_id   INT          PRIMARY KEY,   -- PK and FK combined
    base_salary   DECIMAL(10,2),
    bonus_percent DECIMAL(5,2),
    FOREIGN KEY (employee_id) REFERENCES employees(employee_id)
);

Here the primary key of employee_salary_details is also its foreign key, which automatically guarantees uniqueness. There can be at most one salary-details row per employee.

One-to-Many Relationships

The one-to-many relationship is by far the most common pattern in real-world databases. One row in Table A (the parent or one side) is associated with zero, one, or many rows in Table B (the child or many side). The child table stores a foreign key that references the parent's primary key. Critically, there is no unique constraint on that foreign key column, so multiple child rows can share the same parent value.

The customer-and-orders example above is a textbook illustration. One customer can place many orders, but each order belongs to exactly one customer.

CREATE TABLE customers (
    customer_id INT          PRIMARY KEY,
    full_name   VARCHAR(100) NOT NULL,
    email       VARCHAR(150) UNIQUE NOT NULL,
    city        VARCHAR(80)
);

CREATE TABLE orders (
    order_id    INT          PRIMARY KEY,
    customer_id INT          NOT NULL,        -- FK to customers
    order_date  DATE         NOT NULL,
    total_amount DECIMAL(10,2),
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

Now imagine customer #7 (Alice) places three orders. The customers table has exactly one row for Alice. The orders table has three rows, each with customer_id = 7. Alice's city, email, and name are stored once and only once. If Alice moves, one UPDATE statement fixes the record for all three orders simultaneously. This is the anti-redundancy principle in action.

Other everyday examples of one-to-many relationships include:

  • Departments and employees — one department employs many employees, each employee belongs to one department.
  • Products and reviews — one product can have many reviews, each review covers one product.
  • Blog posts and comments — one post can attract many comments, each comment is attached to one post.
  • Invoices and line items — one invoice contains many line items, each line item belongs to one invoice.

Recognising the one-to-many pattern early in the design process almost always tells you exactly where to put the foreign key: it goes on the many side.

Many-to-Many Relationships

Sometimes the relationship between two entities is genuinely symmetrical and plural on both sides. A student can enroll in many courses, and a course can have many students enrolled in it. An actor can appear in many films, and a film can feature many actors. A product can belong to many categories, and a category can contain many products.

A relational database cannot represent this directly between just two tables — there is nowhere to put a foreign key that would correctly capture the multiplicity on both sides. The solution is a junction table (also called a bridge table, associative table, or linking table). The junction table sits between the two main tables and contains one row for each specific pairing of a record from Table A with a record from Table B. In doing so, it transforms one unrepresentable many-to-many relationship into two perfectly ordinary one-to-many relationships.

CREATE TABLE students (
    student_id INT          PRIMARY KEY,
    full_name  VARCHAR(100) NOT NULL
);

CREATE TABLE courses (
    course_id   INT          PRIMARY KEY,
    course_name VARCHAR(150) NOT NULL,
    credits     TINYINT
);

-- Junction table resolves the many-to-many
CREATE TABLE enrollments (
    student_id    INT  NOT NULL,
    course_id     INT  NOT NULL,
    enrolled_date DATE NOT NULL,
    grade         CHAR(2),
    PRIMARY KEY (student_id, course_id),          -- composite PK
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id)  REFERENCES courses(course_id)
);

With this structure, a student can appear in the enrollments table as many times as they have courses, and a course can appear as many times as it has students — but each specific pairing appears only once (enforced by the composite primary key). The relationship between students and enrollments is one-to-many, and the relationship between courses and enrollments is also one-to-many. Together, they represent the original many-to-many.

Junction Tables and How They Work

A junction table is more than a mechanical necessity — it is itself a full entity in the data model, and it deserves to be treated as such. Every row in a junction table represents a distinct, real-world fact: the fact that this particular student is enrolled in this particular course, that this particular actor appears in this particular film, that this particular product belongs to this particular category.

Because each row represents a meaningful fact, the junction table is an ideal place to record attributes of the relationship itself — information that belongs neither to Table A alone nor to Table B alone, but to the association between them. In the enrollment example above, enrolled_date and grade are perfect examples. The enrollment date is not a property of the student (the same student enrolled on different dates for different courses) and not a property of the course (different students enrolled at different times). It belongs to the specific enrollment. The same logic applies to:

  • A quantity column in an order_items junction table linking orders to products — how many units of that product are in that order.
  • A role column in a film_cast junction table linking films to actors — what character the actor played in that film.
  • A price_at_purchase column in an order_items table — the price of the product at the moment of the order, which may differ from its current price.

The composite primary key on a junction table — typically (table_a_id, table_b_id) — prevents the same pairing from being recorded twice. Some designers add a surrogate integer primary key instead (e.g., enrollment_id), particularly when external systems need to reference individual enrollment rows, but they still enforce a unique constraint on the pair of foreign keys to preserve the uniqueness of each association.

Why Relationships Are Central to Relational Databases

Relationships are not merely a feature of relational databases — they are the defining architectural principle from which everything else flows. Three consequences stand out as especially important.

First, referential integrity. When a foreign key constraint is defined, the database engine takes responsibility for maintaining consistency between the related tables. It will refuse to insert a child row whose foreign key value does not match any existing parent row. It can also be configured to automatically cascade a deletion or update from parent to child rows, or to block the deletion of a parent that still has children, depending on the business rule. This kind of guarantee — enforced at the database level, not just hoped for at the application level — is enormously valuable for data quality.

Second, the ability to join. SQL's JOIN clause is the mechanism by which queries traverse relationships, pulling together columns from multiple tables into a unified result set. Without defined relationships, there would be no principled basis for deciding which rows in two tables belong together. Relationships provide that basis, and joins give you the language to exploit it. Nearly every non-trivial query in a properly normalised database involves at least one join.

-- Retrieve each order alongside the customer's name
SELECT
    o.order_id,
    o.order_date,
    o.total_amount,
    c.full_name     AS customer_name,
    c.city          AS customer_city
FROM orders       AS o
JOIN customers    AS c ON o.customer_id = c.customer_id
ORDER BY o.order_date DESC;

Third, normalization. The formal theory of normalization — first normal form, second normal form, third normal form, and beyond — is essentially the theory of managing dependencies between columns by placing related data into appropriately linked tables. Every normalization step either creates a new table or reorganises columns across tables, and every table it creates needs a relationship back to the table it came from. Relationships are therefore the connective tissue that holds a normalised schema together and allows it to be queried as a coherent whole.

Together, these three properties — referential integrity, joinability, and support for normalization — explain why the relational model has remained the dominant paradigm for storing and querying structured data for more than five decades. No matter how sophisticated a database schema becomes, it is ultimately a collection of tables connected by well-defined relationships, and understanding those relationships at the level of cardinality, keys, and constraints is the prerequisite for designing schemas that are correct, efficient, and maintainable.
