Keys and Identifiers

▶

In any relational database, every piece of data must be traceable, reachable, and unambiguously identifiable. Keys and identifiers are the mechanism that makes this possible. A key is a column — or a set of columns — whose values serve a specific structural role: they uniquely identify rows, enforce business rules, or create navigable links between tables. Without well-designed keys, a database quickly devolves into a collection of disconnected, ambiguous data that cannot be reliably queried or trusted. Understanding keys at a deep level is therefore one of the most important foundations for anyone working with relational data.

What Is a Primary Key?

A primary key is a column or group of columns designated as the definitive identifier for every row in a table. Its job is simple but critical: given any row in the table, the primary key value for that row must be different from the primary key value of every other row. No two rows may share the same primary key, and no row may have a NULL primary key — because NULL means "unknown," and you cannot uniquely identify something whose identity is unknown.

When you declare a primary key on a table, the database engine automatically creates a constraint that enforces both of these rules. If you attempt to insert a row whose primary key value already exists, the database will reject the operation with a constraint violation error. If you attempt to insert a row without providing a primary key value, the database will similarly reject it unless the column is configured to generate its own value automatically.

Consider a simple customers table:

CREATE TABLE customers (
    customer_id   INT          NOT NULL,
    first_name    VARCHAR(50)  NOT NULL,
    last_name     VARCHAR(50)  NOT NULL,
    email         VARCHAR(100),
    PRIMARY KEY (customer_id)
);

Here customer_id is the primary key. Every customer that enters the system receives a unique integer value for this column. The database guarantees that no two customers will ever collide on this value, which means any query using customer_id as a filter will resolve to at most one row — a very powerful guarantee.

A primary key built from a single column is called a simple key. A primary key built from two or more columns working together is called a composite key, which is discussed in detail later. Regardless of how many columns are involved, the rule is the same: the combined value must be unique across all rows and none of the participating columns may be NULL.

Natural Keys vs. Surrogate Keys

Once you decide that a table needs a primary key, the next decision is what that key should be made of. There are two broad strategies: using a natural key or using a surrogate key.

A natural key is a value that already exists in the real-world data being modeled and carries inherent meaning. An employee's national identification number, a product's manufacturer barcode, a country's ISO two-letter code, or a user's email address are all examples of natural keys. The appeal is obvious — the key is self-describing, meaningful to humans, and does not require generating an artificial value.

However, natural keys carry risks. Business data changes. A company might allow customers to change their email address, making an email-based primary key unstable. A product barcode might be reassigned after a product reformulation. A person's name is rarely unique enough and may also change. When a primary key value changes, every foreign key reference to that value in every related table must also change — a potentially enormous and error-prone cascade of updates.

Natural keys may also carry sensitive information. Using a social security number as a primary key, for instance, means that number appears in every join, every log, every related table, and every query — vastly expanding its exposure.

A surrogate key is a value that has no meaning in the real world. It exists only inside the database, manufactured purely to serve as an identifier. The most common form is an auto-incrementing integer: 1, 2, 3, and so on. Modern databases also support universally unique identifiers (UUIDs), which are 128-bit values generated to be globally unique even across separate systems.

-- Auto-incrementing surrogate key (PostgreSQL syntax)
CREATE TABLE products (
    product_id   SERIAL       PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    sku          VARCHAR(50)  UNIQUE NOT NULL
);

-- UUID surrogate key
CREATE TABLE orders (
    order_id    UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_id INT          NOT NULL,
    order_date  DATE         NOT NULL
);

Because a surrogate key carries no business meaning, it never needs to change when business data changes. The customer can update their email, change their name, or move to a new country — the surrogate customer_id remains the same, and every related record continues to point to the right customer without modification.

The trade-off is that surrogate keys require an extra column and offer no self-evident meaning to a human reading the data. In practice, most production databases use surrogate keys for primary keys and then enforce uniqueness on the meaningful business attributes (like email or SKU) through separate unique constraints.

Characteristic Natural Key Surrogate Key
Source Real-world business data System-generated (database or application)
Meaning Carries business meaning No inherent meaning
Stability Can change as business data changes Stable; never needs to change
Privacy risk May expose sensitive information No sensitive information
Human readability Self-describing Opaque without context
Common examples Email, national ID, ISO country code Auto-increment integer, UUID

What Is a Foreign Key?

A foreign key is a column (or set of columns) in one table that references the primary key of another table. It is the concrete mechanism through which relational databases express relationships between entities. Without foreign keys, tables would be isolated silos; with them, a database becomes a connected network of related information.

Imagine an e-commerce database. Orders belong to customers. Rather than duplicating the customer's name, address, and contact details into every order row, you store the customer information once in the customers table and place only the customer's identifier in each order row:

CREATE TABLE orders (
    order_id    INT  NOT NULL PRIMARY KEY,
    customer_id INT  NOT NULL,
    order_date  DATE NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

The column customer_id in the orders table is a foreign key. The database enforces that every value stored in orders.customer_id must correspond to an existing value in customers.customer_id. You cannot create an order for a customer who does not exist.

Foreign keys may also be NULL if the relationship is optional. For example, an order might be placed before it is assigned to a sales representative. If sales_rep_id is nullable, a NULL value legitimately means "not yet assigned" rather than a broken reference.

A single table can carry multiple foreign keys, each linking to a different parent table. An order line item, for instance, might reference both the order it belongs to and the product being purchased:

CREATE TABLE order_items (
    order_item_id INT            NOT NULL PRIMARY KEY,
    order_id      INT            NOT NULL,
    product_id    INT            NOT NULL,
    quantity      INT            NOT NULL,
    unit_price    DECIMAL(10,2)  NOT NULL,
    FOREIGN KEY (order_id)   REFERENCES orders(order_id),
    FOREIGN KEY (product_id) REFERENCES products(product_id)
);

Here order_items has two foreign keys, tying each line item to both its parent order and the specific product involved. This is the relational model working as designed: normalised, non-redundant, and navigable.

Referential Integrity

Referential integrity is the database's guarantee that every foreign key value either points to a real, existing row in the referenced table or is deliberately NULL. When referential integrity is maintained, you will never find an order that references a customer who doesn't exist, or an order item linked to a product that was deleted from the system. Such "orphaned" records are a form of data corruption that produces wrong query results, failed report generation, and difficult-to-diagnose application bugs.

Databases enforce referential integrity through referential actions — rules that define what happens when a referenced row is updated or deleted. The most common options are:

  • RESTRICT / NO ACTION: The operation is blocked. If you try to delete a customer who has existing orders, the database refuses the deletion entirely. This is the safest and most common default behaviour.
  • CASCADE: The operation propagates automatically. Deleting a customer also deletes all of that customer's orders. This is powerful but potentially dangerous if applied without careful thought.
  • SET NULL: When the referenced row is deleted or its key is updated, the foreign key value in child rows is set to NULL. This is appropriate when the relationship is optional.
  • SET DEFAULT: Similar to SET NULL, but replaces the foreign key value with a predefined default rather than NULL.
-- Cascade deletes to child rows
CREATE TABLE orders (
    order_id    INT  NOT NULL PRIMARY KEY,
    customer_id INT  NOT NULL,
    FOREIGN KEY (customer_id)
        REFERENCES customers(customer_id)
        ON DELETE CASCADE
);

-- Prevent deletion of referenced rows
CREATE TABLE order_items (
    order_item_id INT NOT NULL PRIMARY KEY,
    product_id    INT NOT NULL,
    FOREIGN KEY (product_id)
        REFERENCES products(product_id)
        ON DELETE RESTRICT
);

Violations of referential integrity are a significant data quality issue. When data is loaded from external sources, migrated from legacy systems, or modified through backdoor processes that bypass the database constraints, orphaned records can accumulate. Before running analytical queries across related tables, it is wise to check whether foreign key constraints are actually enforced and whether historical data is clean. A JOIN against a corrupted dataset will silently return incomplete or misleading results.

Unique Constraints and Unique Identifiers

A unique constraint is a rule applied to a column or group of columns that prohibits duplicate values. It is similar to a primary key, but with two important distinctions.

First, a table can have only one primary key, but it can have many unique constraints — one per column or group of columns that must be distinct across all rows. Second, in most database systems, a column under a unique constraint (but not a primary key) can contain NULL values, and typically multiple NULLs are permitted because NULL is not considered equal to another NULL.

Returning to the products example: the surrogate key product_id is the primary key, but the business also requires that no two products share the same stock-keeping unit code (SKU). A unique constraint enforces this:

CREATE TABLE products (
    product_id   SERIAL       PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    sku          VARCHAR(50)  NOT NULL UNIQUE,
    price        DECIMAL(10,2)
);

The UNIQUE keyword on the sku column tells the database to reject any insert or update that would create a duplicate SKU. This is a business rule enforced at the data layer, which is far more reliable than trying to enforce it only in application code.

Unique constraints also serve an important role in query performance. Databases automatically create an index to support a unique constraint, so lookups on that column become very fast — the engine can scan the index in logarithmic time rather than scanning every row in the table.

Unique identifiers in the broader sense — values that uniquely distinguish one record from another — are the conceptual backbone of data quality. They make it possible to join tables accurately, deduplicate records, and trace a single entity across multiple systems. An analyst who cannot identify a unique, reliable key for a dataset will struggle to merge it correctly with other data sources.

Composite Keys

A composite key is a primary key (or unique constraint) that spans two or more columns. No single column alone is sufficient to uniquely identify a row; it is the combination of values across all participating columns that must be unique.

The most natural home for composite keys is a junction table (also called a bridge table or associative entity), which represents a many-to-many relationship. Consider a course enrollment system where a student can enroll in many courses, and each course can have many students. The relationship itself lives in a junction table:

CREATE TABLE enrollments (
    student_id  INT  NOT NULL,
    course_id   INT  NOT NULL,
    enrolled_on DATE NOT NULL,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id)  REFERENCES courses(course_id)
);

Here neither student_id alone nor course_id alone is unique — a student appears in many rows (once per course) and a course appears in many rows (once per student). But the pair (student_id, course_id) is unique: a given student can only be enrolled in a given course once. The composite primary key enforces exactly this rule.

Composite keys work correctly but introduce complexity. Any foreign key that references a composite primary key must itself be composite — it must include all the same columns in the same order. Queries involving composite key joins are longer to write and slightly harder to read than those using a single integer.

For these reasons, many database designers add a surrogate key to junction tables and demote the natural composite columns to a unique constraint:

CREATE TABLE enrollments (
    enrollment_id INT  NOT NULL PRIMARY KEY,  -- surrogate
    student_id    INT  NOT NULL,
    course_id     INT  NOT NULL,
    enrolled_on   DATE NOT NULL,
    UNIQUE (student_id, course_id),           -- composite unique, not PK
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id)  REFERENCES courses(course_id)
);

This approach preserves the uniqueness guarantee while giving other tables a simple, single-column integer to reference if they ever need to link to a specific enrollment record.

How Keys Link Data Across Tables

The entire architecture of a relational database rests on a simple but profound pattern: one table stores its primary key, and another table stores a copy of that key as a foreign key. This arrangement creates a directed link — a relationship — between the two tables that can be traversed in SQL using a JOIN.

Visualise the flow of keys through a small e-commerce schema:

  • The customers table stores customer_id as its primary key.
  • The orders table stores its own order_id as its primary key, and holds customer_id as a foreign key linking back to customers.
  • The order_items table stores its own order_item_id as its primary key, holds order_id as a foreign key linking to orders, and holds product_id as a foreign key linking to products.
  • The products table stores product_id as its primary key.

This chain of keys means that starting from a single customer, you can navigate through their orders, into the individual items in each order, and out to the products those items represent — all using JOIN operations that match primary keys to foreign keys:

SELECT
    c.first_name,
    c.last_name,
    o.order_date,
    p.product_name,
    oi.quantity,
    oi.unit_price
FROM customers      c
JOIN orders         o  ON o.customer_id   = c.customer_id
JOIN order_items    oi ON oi.order_id     = o.order_id
JOIN products       p  ON p.product_id    = oi.product_id
WHERE c.customer_id = 42;

Without correctly designed keys, this query would be impossible or would return garbage results. With properly defined primary and foreign keys, the database engine can optimise the join using indexes built on those key columns, making the query fast even over millions of rows.

Well-designed keys deliver four interconnected benefits:

  • Data integrity: Constraints prevent invalid data from entering the system in the first place, ensuring that every relationship refers to something real.
  • Reduced redundancy: Information is stored once, in the table that owns it, and referenced everywhere else by key. Changes made in one place propagate correctly to all queries rather than requiring updates scattered across multiple tables.
  • Query correctness: When you join on a primary key–foreign key pair, you know that every match is intentional and unambiguous — not an accidental collision on a non-unique value.
  • Performance: Indexes created automatically on primary and foreign keys dramatically accelerate lookups and joins, keeping queries responsive as data volumes grow.

Investing time in understanding and correctly designing keys is one of the highest-leverage activities in database work. Every query you write, every report you generate, and every analysis you conduct depends on the accuracy and reliability of the underlying key structure.
