Filtering Data with WHERE

▶

When you query a database, you rarely want every single row in a table. A table holding customer orders might contain millions of records, but you might only care about orders placed last month, or only orders from a specific region, or only orders above a certain dollar amount. The WHERE clause is the primary tool SQL gives you for this job. It acts as a filter that the database engine applies to every row it reads, keeping only the rows that satisfy a condition you define and discarding the rest. Understanding WHERE — and the rich set of operators and keywords that work inside it — is one of the most important skills in practical SQL work.

The basic syntax of a query with a WHERE clause follows this pattern:

SELECT column(s)
FROM table_name
WHERE condition;

The database engine processes this in a specific logical order: it identifies the table named in FROM, then evaluates the WHERE condition against each row one at a time, and finally returns only the columns listed in SELECT for the rows where the condition is TRUE. Rows where the condition is FALSE — or where it cannot be determined — are silently dropped from the result. One important and often overlooked point is that the WHERE clause can reference any column in the table, not only the columns you chose to display in SELECT. For example, you can filter on a last_login date column without including that column in your output.

The following sections explore every major category of filtering you will encounter in day-to-day SQL work.

Comparison Operators

The most fundamental building blocks of a WHERE condition are the comparison operators. They let you test whether a column value stands in a particular relationship to another value. The standard set of operators is:

Operator Meaning Example
= Equal to WHERE department = 'Sales'
<> or != Not equal to WHERE status <> 'Cancelled'
> Greater than WHERE salary > 60000
< Less than WHERE quantity < 10
>= Greater than or equal to WHERE rating >= 4
<= Less than or equal to WHERE age <= 65

The equal operator (=) is used for exact matches. The query below retrieves every employee whose department is exactly Sales:

SELECT first_name, last_name, department
FROM employees
WHERE department = 'Sales';

The not-equal operators (<> and !=) are interchangeable in most database systems. They return every row where the column does not match the value. For instance, to see all orders that are still active — meaning not cancelled — you would write:

SELECT order_id, customer_id, order_date
FROM orders
WHERE status <> 'Cancelled';

Filtering Text Data

When comparing against text (string) values, you must wrap the literal value in single quotes. Writing WHERE city = 'Chicago' is correct SQL. Writing WHERE city = Chicago without quotes will cause the database to interpret Chicago as a column name, which will produce an error if no such column exists, or silently produce wrong results if it does. This is one of the most common beginner mistakes.

For flexible text matching, SQL provides the LIKE operator paired with two wildcard characters:

  • % — matches any sequence of zero or more characters.
  • _ — matches exactly one character.

Some practical examples:

-- Find all customers whose last name starts with 'Sm'
SELECT customer_id, last_name
FROM customers
WHERE last_name LIKE 'Sm%';

-- Find all products with exactly five characters in their code,
-- where the code starts with 'A' and ends with 'Z'
SELECT product_code, product_name
FROM products
WHERE product_code LIKE 'A___Z';

-- Find all email addresses from the gmail domain
SELECT email
FROM users
WHERE email LIKE '%@gmail.com';

When you want to test a column against several specific text values at once, the IN operator is far more readable than stringing together multiple OR conditions:

-- Instead of this:
WHERE country = 'USA' OR country = 'Canada' OR country = 'Mexico'

-- Write this:
SELECT customer_name, country
FROM customers
WHERE country IN ('USA', 'Canada', 'Mexico');

Both forms produce identical results, but IN is cleaner and easier to maintain when the list grows.

Filtering Numeric Data

Numeric values in a WHERE condition are written without quotes. Writing WHERE age = 30 is correct; writing WHERE age = '30' wraps the number in a string literal. While some databases will silently convert the string to a number, others will not, and you will introduce a type mismatch. Always write numbers without quotes to be explicit.

The BETWEEN operator provides a clean way to test whether a value falls within an inclusive range. Both boundary values are included in the result:

-- Returns employees with a salary of 40000, 80000, or anything in between
SELECT employee_id, first_name, salary
FROM employees
WHERE salary BETWEEN 40000 AND 80000;

The inequality operators are especially powerful for business rules and threshold-based filtering. Consider an inventory system where you want to flag products running low:

SELECT product_name, quantity_on_hand
FROM inventory
WHERE quantity_on_hand < 10;

Or a sales report that only cares about high-value transactions:

SELECT order_id, customer_id, total_amount
FROM orders
WHERE total_amount >= 1000;

Combining Conditions with AND and OR

Real queries often need to apply more than one condition at the same time. SQL provides two logical operators for this: AND and OR.

AND requires every connected condition to be TRUE for a row to be included. Think of it as narrowing your results further. The query below returns only employees who are both in the Sales department and were hired after 2020:

SELECT first_name, last_name, department, hire_year
FROM employees
WHERE department = 'Sales'
  AND hire_year > 2020;

OR requires at least one condition to be TRUE. It broadens results. The query below returns employees located in Austin or Denver (or both, if that were somehow possible):

SELECT first_name, last_name, city
FROM employees
WHERE city = 'Austin'
   OR city = 'Denver';

When you mix AND and OR in the same WHERE clause, you must be careful about operator precedence. AND is evaluated before OR, just as multiplication is evaluated before addition in arithmetic. Without parentheses, this query does not do what it appears to do at first glance:

-- POTENTIALLY MISLEADING: AND binds tighter than OR
SELECT *
FROM employees
WHERE department = 'Sales'
   OR department = 'Marketing'
  AND hire_year > 2020;

The database reads this as: department = 'Sales' OR (department = 'Marketing' AND hire_year > 2020). It returns all Sales employees regardless of hire year, plus only post-2020 Marketing employees. If you wanted all Sales or Marketing employees hired after 2020, you must use parentheses:

-- CORRECT: parentheses enforce the intended grouping
SELECT *
FROM employees
WHERE (department = 'Sales' OR department = 'Marketing')
  AND hire_year > 2020;

Always use parentheses when combining AND and OR. It eliminates ambiguity and makes your intent clear to anyone who reads the query later.

Excluding Results with NOT

The NOT keyword negates a condition. It can be paired with IN, LIKE, and BETWEEN to flip their behavior from inclusion to exclusion.

NOT IN excludes every row whose column value appears in the specified list:

-- Returns all records except those with a status of 'Closed' or 'Cancelled'
SELECT order_id, customer_id, status
FROM orders
WHERE status NOT IN ('Closed', 'Cancelled');

NOT LIKE excludes rows where the column matches the given pattern. This is useful for filtering out test or placeholder data:

-- Exclude all email addresses from the internal test domain
SELECT user_id, email
FROM users
WHERE email NOT LIKE '%@test.com';

NOT BETWEEN excludes a range, returning only rows that fall outside the specified boundaries:

-- Returns products priced below $10 or above $50 (excludes $10–$50 range)
SELECT product_name, price
FROM products
WHERE price NOT BETWEEN 10 AND 50;

Handling NULL Values in Filtering

NULL is SQL's way of representing a missing, unknown, or inapplicable value. It is not zero, not an empty string, and not any other concrete value — it simply means no value exists here. This has a critical implication for filtering: you cannot use = or <> to test for NULL.

Consider what happens when you try:

-- This will NOT return rows where email is NULL
SELECT customer_id
FROM customers
WHERE email = NULL;

The expression email = NULL evaluates to unknown (not TRUE and not FALSE) for every row, including rows where email is actually NULL. Because the database only includes rows where the condition evaluates to TRUE, this query returns zero rows — which is almost never what you intended.

The correct approach is to use the dedicated IS NULL and IS NOT NULL keywords:

-- Find all customers who have not provided an email address
SELECT customer_id, first_name, last_name
FROM customers
WHERE email IS NULL;
-- Find all customers who do have a phone number on file
SELECT customer_id, first_name, phone_number
FROM customers
WHERE phone_number IS NOT NULL;

NULL awareness becomes especially important when you combine conditions. If a column contains NULL values, those rows will be silently excluded from any condition using =, <>, >, LIKE, BETWEEN, or IN — even if you might logically expect them to be included. For example, WHERE bonus <> 0 will not return rows where bonus is NULL, because the comparison yields unknown. If you want to include NULL rows as well, you must explicitly add OR bonus IS NULL to your condition.

Putting all of these concepts together, here is a more complete real-world example. Suppose a sales manager wants a list of active, high-value customers based in North America who have a verified email address:

SELECT customer_id, first_name, last_name, email, country, lifetime_value
FROM customers
WHERE country IN ('USA', 'Canada', 'Mexico')
  AND lifetime_value >= 5000
  AND status <> 'Inactive'
  AND email IS NOT NULL
  AND email NOT LIKE '%@test.com';

Each condition independently filters the rows, and because they are joined by AND, only rows satisfying every single condition appear in the result. This is the power of the WHERE clause: the ability to compose multiple, precise conditions into a single, expressive filter that isolates exactly the data you need.

NotesEncourage students to test NULL behavior hands-on, as it is frequently misunderstood. A good exercise is to have them run a query with = NULL and then with IS NULL against the same table and compare the row counts. Emphasize parentheses with AND/OR combinations as a best practice habit, not just an edge case.
